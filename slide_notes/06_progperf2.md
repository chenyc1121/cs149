## 2. 共享地址空间的硬件实现

### 2.1 现代计算机的内存层次结构

单核处理器内部已经包含多级缓存：

```
Core 1

L1 Cache (32 KB)  -->  L2 Cache (256 KB)  -->  L3 Cache (20 MB)  -->  DRAM (32 GB)

... (多核共享 L3 和 DRAM)
```

"将地址 X 处的值加载到寄存器 R0"这一指令，可能涉及多个数据缓存的复杂操作序列和对 DRAM 的访问。

### 2.2 Intel Core i7 环形互联 (Ring Interconnect)

以 Intel Core i7 (Kaby Lake) 四核处理器为例：

- 四个核心 + 内存控制器 + 集成 GPU 通过**环形总线 (Ring Bus)** 连接
- 四条环分别承载不同类型的信息：
  - 请求 (request)
  - 监听 (snoop)
  - 确认 (ack)
  - 数据 (data, 32 字节)
- 六个互联节点：四个 L3 缓存"切片" (slices) + 系统代理 + 图形
- L3 每个 bank 与环形总线连接两次
- 理论峰值带宽（3.4 GHz 下各核访问本地 slice）：约 435 GB/sec

### 2.3 SUN Niagara 2: 交叉开关互联 (Crossbar Interconnect)

- 八核处理器，使用**交叉开关 (Crossbar Switch)**
- 交叉开关面积约等于一个核心的面积
- **交叉开关 = 所有核心直接连接所有其他核心**

### 2.4 非均匀内存访问 (NUMA)

在现代多插槽 (multi-socket) 配置中：

- 不同处理器核心访问同一内存位置的**延迟可能不同**
- 不同 CPU 核心到同一位置的**带宽也可能不同**
- 即使在单插槽系统中也存在 NUMA 行为（不同缓存 slice 距离各核心的距离不同）

### 2.5 共享地址空间模型小结

**通信抽象：**
- 线程通过读写共享地址空间中的变量进行通信
- 线程操作同步原语：锁、原子操作等
- 是单处理器编程的逻辑扩展

**硬件要求：**
- 任何处理器都能从任何地址加载和存储
- 扩展到大量处理器时代价高（这是高核心数处理器昂贵的原因之一）
- 但 NUMA 实现需要**考虑局部性**来进行性能优化

## 3. 消息传递模型 (Message Passing)

### 3.1 模型抽象

在消息传递模型中：

- 线程在**各自的私有地址空间**内运行
- 线程通过**发送/接收消息**进行通信
  - `send()`：指定接收者、传输的缓冲区、可选的消息标识符 (`tag`)
  - `recv()`：指定发送者、存储数据的缓冲区、可选的消息标识符 (`tag`)
- **发送消息是线程之间交换数据的唯一方式**

```
Thread 1:  send(X, 2, my_msg_id)     // 将本地变量 X 的内容发送给线程 2，标记为 "my_msg_id"
Thread 2:  recv(Y, 1, my_msg_id)     // 从线程 1 接收标记为 "my_msg_id" 的消息，存入本地变量 Y
```

### 3.2 实现层面

- 硬件不需要为所有处理器实现单一共享地址空间
- 只需要提供节点间通信消息的机制
- 可以将**商用系统连接起来**形成大规模并行机（集群和超级计算机的编程模型）
- 典型实例：通过 Infiniband 网络连接的工作站集群

## 4. 网格求解器的消息传递实现

### 4.1 问题回顾

N x N 网格求解器应用：
1. 并行更新所有红色格子
2. 完成后更新所有黑色格子（依赖红色格子的结果）
3. 重复直到收敛

### 4.2 消息传递配置示例

以两台机器的集群为例，每台有自己的处理器、缓存和内存，通过网络连接。

### 4.3 数据分区与幽灵单元 (Ghost Cells)

- 网格数据被划分为四个分区，每个分区位于一个线程的私有地址空间中
- **幽灵单元 (Ghost Cells)**：从远程地址空间复制到本地的网格单元，其信息由其他线程"拥有"
- 例如：线程 1 和线程 3 在完成红色单元处理后，分别向线程 2 发送一行数据

### 4.4 消息传递求解器伪代码

```c
int N;
int tid = get_thread_id();
int rows_per_thread = N / get_num_threads();
float* localA = allocate(rows_per_thread+2, N+2);

void solve() {
  bool done = false;
  while (!done) {
    float my_diff = 0.0f;

    // 步骤 1: 向邻居线程发送和接收幽灵行
    if (tid != 0)
       send(&localA[1,0], sizeof(float)*(N+2), tid-1, MSG_ID_ROW);
    if (tid != get_num_threads()-1)
       send(&localA[rows_per_thread,0], sizeof(float)*(N+2), tid+1, MSG_ID_ROW);

    if (tid != 0)
       recv(&localA[0,0], sizeof(float)*(N+2), tid-1, MSG_ID_ROW);
    if (tid != get_num_threads()-1)
       recv(&localA[rows_per_thread+1,0], sizeof(float)*(N+2), tid+1, MSG_ID_ROW);

    // 步骤 2: 执行计算 (与共享地址空间版本相同)
    for (int i=1; i<rows_per_thread+1; i++) {
       for (int j=1; j<n+1; j++) {
         float prev = localA[i,j];
         localA[i,j] = 0.2 * (localA[i-1,j] + localA[i,j] + localA[i+1,j] +
                              localA[i,j-1] + localA[i,j+1]);
         my_diff += fabs(localA[i,j] - prev);
       }
    }

    // 步骤 3: 所有线程将局部 diff 发送给线程 0
    // 步骤 4: 线程 0 计算全局 diff，判断是否收敛，广播结果
    if (tid != 0) {
       send(&mydiff, sizeof(float), 0, MSG_ID_DIFF);
       recv(&done, sizeof(bool), 0, MSG_ID_DONE);
    } else {
       float remote_diff;
       for (int i=1; i<get_num_threads()-1; i++) {
          recv(&remote_diff, sizeof(float), i, MSG_ID_DIFF);
          my_diff += remote_diff;
       }
       if (my_diff/(N*N) < TOLERANCE)
         done = true;
       for (int i=1; i<get_num_threads()-1; i++)
         send(&done, sizeof(bool), i, MSG_ID_DONE);
    }
  }
}
```

### 4.5 消息传递示例的关键要点

- **计算**：数组索引相对于本地地址空间
- **通信**：通过发送和接收消息执行；**批量传输**：一次通信整行数据
- **同步**：通过发送和接收消息来实现（可以用消息实现互斥锁、屏障、标志位等）

## 5. 同步 vs 异步消息传递

### 5.1 同步（阻塞）send/recv

- **`send()`**：调用返回时，发送方已收到确认——消息数据已存在于接收方的地址空间中
- **`recv()`**：调用返回时，接收到的消息数据已复制到接收方地址空间，确认已发回发送方

**执行流程：**

```
发送方: 调用 SEND(foo)
        -> 将数据从缓冲区 'foo' 复制到网络缓冲区
        -> 发送消息
        -> (等待...)
        -> 接收确认
        -> SEND() 返回

接收方: 调用 RECV(bar)
        -> 接收消息
        -> 将数据复制到缓冲区 'bar'
        -> 发送确认
        -> RECV() 返回
```

### 5.2 同步消息传递的死锁问题

原始的网格求解器使用同步 send/recv 时存在**严重的死锁问题**。

**死锁原因分析：**
- 如果所有线程都先 `send` 再 `recv`
- 每个线程的 `send` 都在等待对方的 `recv` 来接收数据
- 但对方也在等待自己先 `send` 完成
- 互相等待，形成死锁

**修复方案：交替 send/recv 顺序**

```c
if (tid % 2 == 0) {
   // 偶数编号线程：先 sendDown，再 recvDown，先 sendUp，再 recvUp
   sendDown(); recvDown();
   sendUp();   recvUp();
} else {
   // 奇数编号线程：先 recvUp，再 sendUp，先 recvDown，再 sendDown
   recvUp();   sendUp();
   recvDown(); sendDown();
}
```

时间线示意：

```
T0:  send -> recv
T1:        recv -> send
T2:  send -> recv
T3:        recv -> send
T4:  send -> recv
T5:        recv -> send
     ------------------------------> 时间
```

偶数编号线程先发送后接收，奇数编号线程先接收后发送，避免了循环等待。

### 5.3 非阻塞异步 send/recv

**`send()`：调用立即返回**
- 提供给 `send()` 的缓冲区在消息处理完成前**不能被调用线程修改**
- 调用线程可以在等待消息发送期间执行其他工作
- 使用 `checksend()` 检查发送状态

**`recv()`：发送接收意图后立即返回**
- 使用 `checkrecv()` 检查实际接收状态
- 调用线程可以在等待消息接收期间执行其他工作

**执行流程：**

```
发送方: 调用 SEND(foo)           -> 返回句柄 h1（立即返回）
        (并发) 将数据从 'foo' 复制到网络缓冲区，发送消息
        调用 CHECKSEND(h1)       -> 检查是否发送完成，安全后可修改 'foo'

接收方: 调用 RECV(bar)           -> 返回句柄 h2（立即返回）
        (并发) 消息库将数据复制到 'bar'
        调用 CHECKRECV(h2)       -> 检查是否接收完成，安全后可访问 'bar'

红色文字 = 与应用程序线程并发执行
```

## 6. 广义的"通信"概念

通信不仅仅是机器之间的消息传递，还包括：

- 芯片上核心之间的通信
- 核心与其缓存之间的通信
- 核心与内存之间的通信

### 6.1 将并行系统视为扩展的内存层次结构

从一个处理器的视角看，数据访问按以下顺序进行：

```
寄存器 --> 本地 L1 --> 本地 L2 --> L3 缓存 --> 本地内存 --> 远程内存(1跳) --> 远程内存(N跳)
|_____ 低延迟、高带宽、小容量 _____||_____ 高延迟、低带宽、大容量 _____|
```

**核心思想：**
- 本地缓存未命中的访问会导致与下一层级的通信
- **在所有层级上管理局部性以减少通信量都非常重要**

### 6.2 CPU 到内存的通信示例

```
处理器发出加载指令
  -> L1 缓存查找
  -> L2 缓存查找
  -> 向内存发送请求
  -> 通过内存总线从内存传输缓存行     <-- 通信时间（缓存行在内存总线上的传输时间）
  -> 将值传输到处理器寄存器
  -> 总的内存访问延迟
```

## 7. 算术强度 (Arithmetic Intensity)

### 7.1 定义

```
算术强度 = 计算量 (如指令数) / 通信量 (如字节数)

倒数 "1/算术强度" = 通信-计算比 (Communication-to-Computation Ratio)
```

- **高算术强度 = 低通信-计算比** —— 这是我们追求的目标
- 现代并行处理器的计算能力与可用带宽之比非常高，因此需要高算术强度才能高效利用

### 7.2 带宽受限执行回顾

在带宽受限的场景中：

- 指令完成速率由内存提供数据的速率决定
- 内存 100% 时间内都在传输数据——无法更快
- 核心利用率仅取决于**指令吞吐量**和**内存吞吐量**的比值
- 与内存延迟或未完成的内存请求数量无关

**关键洞察题：**
1. 如何从图中判断内存总线已完全利用？
2. 如何说明更高的内存延迟（假设请求流水线化且总线带宽不变）？
3. 如果内存总线带宽增加，图会如何变化？
4. 如果数学指令与加载指令的比例大幅提高，处理器是否还会停顿？

## 8. 固有通信 vs 人为通信

### 8.1 固有通信 (Inherent Communication)

- 并行算法**必须发生**的通信
- 是算法本身**根本属性**的通信
- 例如：网格求解器中发送幽灵行的通信

### 8.2 降低固有通信：通过更好的任务分配

**1D 块状分配 (Blocked Assignment)：**

```
N x N 网格，P 个处理器
每处理器计算量: 约 N^2 / P
每处理器通信量: 约 2N
算术强度: ∝ N / P    (与 P 成反比，随处理器数增加而降低)
```

**1D 交错分配 (Interleaved Assignment)：**

```
每处理器计算量: 约 N^2 / P
每处理器通信量: 约 2N
算术强度更低，约为 1/2
```

**2D 块状分配 (最佳方案)：**

```
N x N 网格，P 个处理器
每处理器计算量: N^2 / P
每处理器通信量: 约 4 * (N / √P)
算术强度: ∝ N / √P
```

- 渐进意义上比 1D 分配**通信扩展性更好**
- 通信成本随 P **亚线性增长**
- 分配方式捕获了算法的**2D 局部性**

### 8.3 人为通信 (Artifactual Communication)

- **人为通信**：所有其他通信
- 源于系统实现的实际细节
- 包括：
  - 缓存行为导致的额外通信
  - 最小传输粒度（如必须传输整个 64 字节缓存行而不是仅需要的 4 字节浮点数）
  - 有限缓存容量导致的容量缺失

## 9. 人为通信的实例：缓存行为

### 9.1 网格求解器行优先遍历的问题

**场景设定：**
- N x N 网格，行优先存储
- 缓存行大小：4 个网格元素
- 缓存容量：24 个网格元素（6 条缓存行）

**问题：**
- 处理第一行时缓存了部分数据（蓝色标记）
- 处理完第一行时缓存中有一些数据
- 但处理到第二行第一个输出元素时，之前访问过的 `(0,1)`, `(1,1)`, `(2,1)`, `(0,2)`, `(2,2)` 等元素已经不在缓存中
- **结果：每4个输出元素需要加载3条缓存行**（其中很多数据没被用到就被逐出了）

### 9.2 人为通信的常见形式

1. **最小传输粒度问题**：加载一个 4 字节 float 值，但必须传输整个 64 字节缓存行（多出 16 倍通信）

2. **不必要的通信**：存储 16 个连续的 4 字节 float，导致整个 64 字节缓存行被加载、完全覆盖、再存回内存（2 倍开销——加载是不必要的，因为整行都被覆盖了）

3. **有限复制容量**：相同的缓存行被多次传输给处理器，因为缓存太小无法在两次访问之间保留（容量缺失）

## 10. 降低通信的技术

### 10.1 分块 (Blocking) 改善时间局部性

通过**重新排列计算顺序**来减少容量缺失：

```
行优先遍历：每 4 个输出元素需要加载 3 条缓存行
分块遍历：  每 6 个输出元素需要加载 2 条缓存行
```

**分块策略：**
- 将网格划分为若干块 (blocks)
- 逐块处理，处理完一块再处理下一块
- 块内的数据可以留在缓存中被重复利用

### 10.2 循环融合 (Loop Fusion)

**融合前（模块化但性能差）：**

```c
void add(int n, float* A, float* B, float* C) {
    for (int i=0; i<n; i++)
       C[i] = A[i] + B[i];
}
void mul(int n, float* A, float* B, float* C) {
    for (int i=0; i<n; i++)
       C[i] = A[i] * B[i];
}

float* A, *B, *C, *D, *E, *tmp1, *tmp2;
// 计算 E = D + ((A + B) * C)
add(n, A, B, tmp1);       // 2 次加载, 1 次存储 per math op (AI = 1/3)
mul(n, tmp1, C, tmp2);    // 2 次加载, 1 次存储 per math op (AI = 1/3)
add(n, tmp2, D, E);       // 2 次加载, 1 次存储 per math op (AI = 1/3)
// 整体算术强度 = 1/3
```

**融合后（高性能）：**

```c
void fused(int n, float* A, float* B, float* C, float* D, float* E) {
    for (int i=0; i<n; i++)
       E[i] = D[i] + (A[i] + B[i]) * C[i];
}
// 计算 E = D + (A + B) * C
fused(n, A, B, C, D, E);
// 4 次加载, 1 次存储 / 3 个 math ops (AI = 3/5)
// 整体算术强度 = 3/5 >> 1/3
```

**为什么融合版本性能更好？**
- 融合前的版本需要将中间结果 (`tmp1`, `tmp2`) 写入内存再重新加载
- 融合后中间结果留在寄存器中，避免了不必要的内存访问
- 虽然融合前的代码更加模块化（类似 numPy 风格），但代价是更多的内存通信

### 10.3 通过数据共享提高算术强度

**核心思想：共享数据——将操作相同数据的任务放在一起**

- 将操作相同数据结构的线程安排在同一时间、同一处理器上执行
- 可以减少固有通信

## 11. 竞争 (Contention)

### 11.1 竞争的概念

- 资源有一定的操作吞吐量（单位时间的事务数）——如内存、通信链路、服务器等
- **竞争**发生在一小段时间内对某个资源有大量请求时（该资源成为"热点"）

### 11.2 类比：办公室办公时间

**场景：** 教授办公时间 3:00-3:20pm，无需预约

- 操作：教授帮助学生解答问题
- 执行资源：教授本人
- 步骤：
  1. 学生从咖啡馆走到教授办公室（5 分钟）
  2. 学生排队等待（如果需要）
  3. 教授用洞察力回答问题（5 分钟）

**无预约 → 竞争：**
- 学生1：花 10 分钟（走动 5 + 答疑 5，没有等待）
- 学生5：花 23 分钟（走动 5 + 等待 13 + 答疑 5）
- 问题：共享资源的竞争导致整体操作时间延长

**有预约 → 无竞争：**
- 学生1（3pm 预约）和学生2（4pm 预约）：各花 10 分钟
- 通过时间分散避免了竞争

### 11.3 减少竞争的通信结构

**扁平通信 (Flat Communication)：**
- 潜在高竞争
- 但无竞争时延迟低

**树状通信 (Tree-Structured Communication)：**
- 降低竞争
- 但无竞争时延迟更高（多跳）

**示例：更新共享变量**——如果用树状归约，可以避免所有线程同时访问同一个变量。

### 11.4 分布式工作队列减少竞争

**问题：** 单一共享工作队列的竞争

**解决方案：**
- 每个工作线程有**自己的工作队列**
- 从自己的工作队列取数据、放数据（当所有处理器都有工作时无竞争）
- 当本地工作队列为空时：从随机的其他工作队列**窃取 (steal)** 工作
  - 此时同步是可以接受的，因为线程反正会空闲

```
       T1      T2      T3      T4
    [queue] [queue] [queue] [queue]
       |       |       |       |
       多个工作队列（通常每个工作线程一个）
       
       当 T2 本地队列为空：从 T4 窃取工作 <-- steal!
```

## 12. 降低通信成本：总结

### 12.1 降低通信开销

- 发送更少的消息，使每条消息更大（摊薄开销）
- 将多个小消息合并为大消息

### 12.2 降低通信延迟

- **应用编写者**：重构代码以利用局部性
- **硬件实现者**：改进通信架构

### 12.3 减少竞争

- 复制被竞争的资源（如本地副本、细粒度锁）
- 错开对被竞争资源的访问

### 12.4 增加通信/计算重叠

- **应用编写者**：使用异步通信（如异步消息传递）
- **硬件实现者**：流水线、多线程、预取、乱序执行
- **要求**：应用程序中需要额外的并发性（比执行单元数量更多的并发）

## 13. 性能分析策略

### 13.1 黄金法则

**始终、始终、始终先尝试最简单的并行解决方案，然后测量性能，看看自己处于什么水平。**

### 13.2 核心分析问题

确定程序性能受限于：
- **计算**？（指令吞吐量不足）
- **内存带宽**（或内存延迟）？
- **同步**？

### 13.3 Roofline 模型

Roofline 模型描述了一个具有给定算术强度的程序可达到的最大指令吞吐量：

- **X 轴**：算术强度（不同点对应不同程序的算术强度）
- **Y 轴**：最大可获得的指令吞吐量

**两个区域：**

```
  |
  |  水平区域：计算受限 (Compute Limited)
  |  =============================
  |                                   /
  |                                  /
  |  对角线区域：内存带宽受限        /
  |  (Memory Bandwidth Limited)    /
  |                              /
  |____________________________/
                 算术强度
```

- **水平天花板**：受处理器峰值计算能力限制
- **对角线天花板**：受内存带宽与算术强度的乘积限制

**Roofline 模型中的优化区域：**
- 使用不同级别的优化基准测试（例如使用/不使用 SIMD 指令的最佳性能）
- 可以清晰看出程序处于哪个瓶颈区域，以及优化潜力

### 13.4 建立高水位线 (High Watermarks)

这是一套非常有用的性能诊断技巧：

| 实验 | 方法 | 诊断意义 |
|------|------|----------|
| **增加数学运算** | 添加非内存指令（纯计算） | 执行时间是否随操作计数线性增加？如果是，说明代码受指令速率限制 |
| **消除数据访问局部性需求** | 将所有数组访问改为 `A[0]` | 代码快了多少？建立了改善数据访问局部性的收益上限 |
| **消除同步开销** | 移除所有原子操作和锁 | 代码快了多少？建立了减少同步开销的收益上限 |
| **消除计算** | 移除大部分数学运算，但加载相同数据 | 执行时间减少多少？如果不多，说明是内存瓶颈 |

**重要提示：** 计算、内存访问和同步几乎永远不会完美重叠，因此整体性能很少会完全由计算或带宽或同步单独决定。但以上程序修改带来的性能变化敏感度，可以很好地指示主要成本在哪里。

### 13.5 使用性能分析工具

**现代处理器都有低级别事件"性能计数器" (Performance Counters)：**
- 记录指令完成数、时钟周期、L2/L3 缓存命中/缺失、从内存控制器读取的字节数等

**工具示例：**
- Intel Performance Counter Monitor (PCM)：提供 C++ API 访问硬件性能计数器
- Intel VTune
- PAPI
- oprofile

```cpp
PCM *m = PCM::getInstance();
SystemCounterState begin = getSystemCounterState();
// 待分析的代码放在这里
SystemCounterState end = getSystemCounterState();

printf("Instructions per clock: %f\n", getIPC(begin, end));
printf("L3 cache hit ratio: %f\n", getL3CacheHitRatio(begin, end));
printf("Bytes read: %d\n", getBytesReadFromMC(begin, end));
```

**注意：** 操作系统活动监视器显示的"CPU 使用率百分比"对性能优化帮助不大。

## 14. 问题规模与扩展性

### 14.1 如何衡量一个好的设计？

**性能衡量指标：**

- **绝对性能**：通常以墙钟时间 (wall clock time) 衡量，或每秒操作数
- **加速比 (Speedup)**：
  ```
  加速比 = 顺序程序执行时间 / P 个处理器上的执行时间
        = P 个处理器上的每秒操作数 / 顺序程序的每秒操作数
  ```
- **效率 (Efficiency)**：
  ```
  效率 = 每单位资源的性能
  例如：每芯片面积的每秒操作数、每美元的每秒操作数、每瓦特的每秒操作数
  ```

### 14.2 正确衡量加速比的基准

**常见陷阱：**
- 用并行算法在单核上运行来衡量"基准"加速比（这会让加速比看起来很好看）
- 正确做法：应该与**最佳顺序程序**比较

例如：网格求解器的新并行算法收敛可能更慢（需要更多迭代），所以加速比应该与原始顺序算法比较，而不是与并行算法在单核上运行比较。

### 14.3 固定问题规模的陷阱

**场景：** 258 x 258 网格在 32 核机器上运行
- 每个处理器约 310 个网格单元
- 通信-计算比太大
- 结果：**没有性能提升，反而略有性能下降**

**核心原因：**
- 对于该机器来说，问题规模**太小**
- 大规模机器上的小问题不行：通信开销 > 并行收益
- 但可能小问题的绝对运行时间单核上已经足够快

**另一种情况：超级线性加速比 (Super-linear Speedup)**

当有足够多的处理器时，分配给每个处理器的网格块（工作集）可以完全放入缓存中时：
- 单机上工作集可能超出缓存容量造成缓存颠簸
- 多机上每个处理器的工作集小于缓存，反而出现超级线性加速比
- 另一个例子：单机内存不够导致磁盘交换，多机有更多总内存则不会

### 14.4 固定问题 vs 扩展问题规模

**问题规模与机器规模之间的复杂交互：**
- 影响负载均衡、开销、算术强度、数据访问局部性
- 效果可能很显著且依赖应用

**问题太小：**
- 并行开销压倒并行收益（甚至可能导致性能下降）
- 适合小机器的问题规模不适合大机器

**问题太大：**
- 关键工作集可能在小机器中"放不下"（磁盘交换、超出缓存容量、甚至无法运行）
- 大机器工作集放得下但小机器放不下时，可能出现超级线性加速比

**更好的做法：**
- 随着机器规模增长而**扩展问题规模**
- 买更大的机器是为了**计算更多**，而不仅仅是更快地计算同一个问题
- 这样更能反映实际使用场景

### 14.5 扩展性分析要点

回顾 2D 块状分配的算术强度：

```
算术强度 ∝ N / √P

当 N 固定、P 增大时 → 算术强度下降 → 通信开销增大 → 加速比受限
当 N 随 P 一起增长时 → 算术强度可保持不变 → 良好的可扩展性
```

## 16. 性能优化技巧总结

| 原则 | 说明 |
|------|------|
| **测量、测量、再测量** | 永远不要猜测性能瓶颈，用数据说话 |
| **先简单后复杂** | 总是先尝试最简单的并行方案，测量性能后再决定是否需要优化 |
| **建立高水位线** | 确定理论上的最佳性能上限，判断自己离最佳有多远 |
| **识别瓶颈类型** | 是计算受限、同步受限还是带宽受限？ |
| **关注问题规模** | 问题规模是否与机器规模匹配？是否需要扩展问题规模？ |
| **减少固有通信** | 通过更好的任务分配提高算术强度（如 2D 分块优于 1D 分块） |
| **减少人为通信** | 分块计算、循环融合、利用时间和空间局部性 |
| **减少竞争** | 分布式工作队列、树状通信、资源复制 |
| **重叠通信与计算** | 异步消息、流水线、预取、多线程 |
| **使用性能分析工具** | 硬件性能计数器、VTune、PAPI 等 |
