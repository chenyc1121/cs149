## 1. 缓存一致性回顾

### 1.1 缓存一致性问题

现代处理器在本地缓存中复制内存内容，导致不同处理器可能对同一内存位置观测到不同的值。

**核心问题**：缓存的存在使得同一地址 X 的数据在多个处理器的缓存中可能存在不一致的副本。

```
初始: 地址 X 处的值为 0 (write-back cache)

P1: store X=1    → P1 cache: 1, 内存: 0, 其他cache: 0
P1: load Y        → (假设导致 X 被驱逐出 P1 缓存)
P3: load X        → P3 cache: 0 (miss, 从内存读取旧值!)
P3: store X=2     → P3 cache: 2
P2: load X        → P2 cache: 2 (从 P3 获取?)
```

**这是互斥问题吗？能用锁解决吗？**
- **不能！** 锁解决的是不同处理器同时访问同一地址的问题
- 缓存一致性是由数据在本地缓存中复制引起的问题
- 即使只有一个处理器访问，缓存复制本身也会导致不一致

### 1.2 一致性的定义

一个内存系统是**一致的 (coherent)**，当且仅当对于每个内存位置，存在一个假设的串行顺序 (serial order)，使得：

1. **每个处理器发出的内存操作按其发出顺序执行**（单处理器保序）
2. **读操作返回的值是该位置的"最后一次"写入的值**（由串行顺序定义）

```
地址 X 上的操作时序:
P0 write: 5
P1 read:  (5)
P2 read:  (5)
P0 read:  (5)
P1 write: 25
P0 read:  (25)
```

### 1.3 MSI 写回-失效协议

MSI 协议是经典的监听式 (snooping) 缓存一致性协议。

**三种缓存行状态：**
| 状态 | 含义 |
|------|------|
| **I (Invalid)** | 缓存行无效（与单处理器缓存中的含义相同） |
| **S (Shared)** | 缓存行在一个或多个缓存中有效，内存是最新的 |
| **M (Modified)** | 缓存行仅在恰好一个缓存中有效（独占、脏状态） |

**处理器操作（由本地 CPU 触发）：**
- `PrRd` — 读操作
- `PrWr` — 写操作

**总线事务（来自远程缓存）：**
- `BusRd` — 获取缓存行副本，无意修改
- `BusRdX` — 获取缓存行副本，有意修改（同时使其他副本失效）
- `BusWB` — 将脏行写回内存

**MSI 状态转换规则：**
- 读操作将块以 **S (Shared)** 状态获取（即使只有一个缓存的副本）
- 写入前必须获得**独占所有权**
- `BusRdX` 导致其他缓存失效
- 如果在另一个缓存中处于 M 状态，会触发写回 (writeback)
- 即使在 S 状态命中，也需要 `BusRdX` 升级为 M 状态 (upgrade)

**状态转换图：**

```
         PrRd /--
    M ──────────── M
    │  BusRdX/BusWB
    │
    │  PrWr / BusRdX
    ▼
    S ◄─────────── I
    ▲              ▲
    │  PrWr / --   │ BusRd/BusWB
    │              │
    └──────────────┘
      PrWr / BusRdX
      
    (实线 = 处理器发起, 虚线 = 总线发起)
```

### 1.4 MSI 如何满足缓存一致性不变式

1. **单写者多读者 (SWMR) 不变式**：
   - 只有一个缓存可以处于 M 状态，其他所有缓存都会收到失效消息
   - 多个缓存可以同时处于只读的 S 状态

2. **数据值不变式 (写串行化)**：
   - 在 `BusRd` 和 `BusRdX` 时，数据由处于 M 状态的缓存提供
   - 总线将所有事务串行化

```
时间 →
地址 x:
  Read-Write (P0)    Read-Only (P0, P1, P2)
  Read-Write (P1)    Read-Only (P0, P1)
```

### 1.5 MESI 失效协议

**MSI 的低效问题**：即使在应用程序没有任何共享的情况下，MSI 也需要两次总线事务来完成"读后写"的常见模式：
1. `BusRd`：从 I 状态移动到 S 状态
2. `BusRdX`：从 S 状态移动到 M 状态（升级）

**MESI 的改进**：增加 **E (Exclusive Clean)** 状态：
- 缓存行未被修改，但**只有此缓存**持有该行的副本
- 将独占性与行所有权解耦（行不脏，内存中的副本仍然有效）
- 从 E 升级到 M **不需要总线事务**（因为已经独占）

**MESI 状态转换图：**

```
      PrRd / --        PrWr / --
    E ────────── E    E ────────── M
    │                  
    │ PrWr / BusRdX    
    ▼                  
    S ◄──── I          
    │  BusRd / BusWB   
    │                  
    │ PrRd / BusRd     
    │ (no other cache  
    │  asserts shared) 
```

### 1.6 基于目录的缓存一致性

**监听方案的局限**：广播一致性消息以确定行在其他缓存中的状态——**不可扩展**。

**目录方案**：将缓存行状态信息存储在一个集中位置——**目录 (directory)**。

- 目录条目包含该缓存行在所有缓存中的状态信息
- 缓存按需从目录中查找信息
- 通过点对点消息在缓存之间按"需要知道"的原则维护一致性
- 目录（而非总线）作为一致性串行化机制
- 仍需维护 SWMR 和写串行化的不变式

### 1.7 Intel Core i7 中的目录一致性

```
Core 0: L1 Data Cache → L2 Cache
Core 1: L1 Data Cache → L2 Cache
Core 2: L1 Data Cache → L2 Cache  
Core 3: L1 Data Cache → L2 Cache
         ↓                ↓
    共享 L3 Cache (每核心一个 Bank)
         ↓
    Ring Interconnect (环形互连)
```

- **L3 作为集中式目录**：由于 L3 是包含式缓存 (inclusive cache)，L2 中的任何行都保证驻留在 L3 中
- 目录维护包含该行的 L2 缓存列表
- 不再向所有 L2 广播一致性流量，只向包含该行的 L2 发送一致性消息
- Core i7 互连是环形 (ring) 而非总线 (bus)
- 目录维度：P = 4（核心数），M = L3 缓存行数

### 1.8 缓存一致性对程序员的影响

**通信开销 (Communication Overhead)**：

```
        单处理器              多处理器
寄存器    宽度表示访问频率
L1 Cache    ↓ (更高的缺失率)
L2 Cache    ↓ (更高的缺失率)
主存        ↓ (NUMA中可能"缺失")
远程内存    ↓ (额外长延迟)
```

**平均内存访问时间 (AMAT)**：
```
AMAT = Σ (访问频率_i × 访问延迟_i)
AMAT_多处理器 > AMAT_单处理器
```

**Core i7 Xeon 5500 系列数据源延迟（近似）：**
| 数据源 | 延迟 |
|--------|------|
| L1 命中 | ~4 周期 |
| L2 命中 | ~10 周期 |
| L3 命中，行不共享 | ~40 周期 |
| L3 命中，行在另一核心共享 | ~65 周期 |
| L3 命中，行在另一核心被修改 | ~75 周期 |
| 本地 DRAM | ~30 ns (~120 周期) |
| 远程 DRAM | ~100 ns (~400 周期) |

**优化工具**：Intel VTune、Apple Xcode Instruments

### 1.9 伪共享 (False Sharing)

**问题代码：**
```c
// 为每个线程分配一个变量用于本地累积
int myPerThreadCounter[NUM_THREADS];
```

**为什么会慢？** 多个 `int` 变量位于同一缓存行中，不同线程写入各自的计数器时会导致缓存行在不同核心间"乒乓"传递。

**优化方案：**
```c
// 使用填充将每个变量分隔到不同的缓存行
struct PerThreadState {
    int myPerThreadCounter;
    char padding[CACHE_LINE_SIZE - sizeof(int)];
};
PerThreadState myPerThreadCounter[NUM_THREADS];
```

**性能对比（4 核系统，8 线程）：**
| 方案 | 执行时间 |
|------|----------|
| 无填充 | 14.2 秒 |
| 有填充 | 4.7 秒 |

**伪共享的定义**：两个处理器写入不同的地址，但这些地址映射到同一个缓存行。

```
P1 ────┐
        ├── 同一个缓存行 ── "乒乓"效应
P2 ────┘
```

- 这种通信完全是**人为的 (artifactual)**，缓存行 > 4B 导致了这个问题
- 在编写缓存一致架构的程序时必须考虑伪共享

### 1.10 缓存行大小对缺失率的影响

通过模拟 1 MB 缓存在四种应用上的表现：

| 应用 | 主要缺失类型 |
|------|-------------|
| Barnes-Hut | 升级 (Upgrade)、伪共享、真实共享、容量/冲突 |
| Radiosity | 升级、伪共享、真实共享、容量/冲突 |
| Ocean Sim | 升级、伪共享、真实共享、容量/冲突 |
| Radix Sort | 升级、伪共享、真实共享、容量/冲突 |

缺失率随缓存行大小（8B → 256B）变化：
- 冷缺失 (Cold)：大缓存行减少冷缺失
- 容量/冲突缺失：大缓存行通常导致更多冲突
- 真实共享：影响相对稳定
- 伪共享：大缓存行增加伪共享缺失
- 升级 (Upgrade)：S→M 转换

## 2. 内存一致性概述

### 2.1 共享内存的行为

**直觉上**：load 应该返回"最新"写入的值。

**问题**：什么是"最新"？

- **一致性 (Coherence)**：仅涉及**一个**内存地址，定义对该地址的操作顺序
- **一致性 (Consistency)**：涉及**所有**地址的**表观**顺序

**内存一致性定义了**：
- 一个线程执行的内存操作何时对另一个线程可见
- 影响**可编程性**：程序员如何推理程序行为
- 影响**性能**：允许的硬件/软件优化范围
- 影响可**重排序**的空间：为隐藏延迟而重新排序内存操作

### 2.2 谁应该关心中存一致性？

- 想要实现**同步库**的人
- 从事**内核（或驱动）开发**的人
- 实现**无锁数据结构 (lock-free data structures)** 的人

### 2.3 内存一致性 vs. 内存一致性

**内存一致性**定义了对于**同一内存位置**的读写行为的观察要求：
- 所有处理器必须对地址 X 的读写顺序达成一致
- 可以将所有涉及 X 的操作放在一个时间轴上，使所有处理器的观察与该时间轴一致

```
地址 X 上的操作时序:
P0 write: 5
P1 read:  (5)
P2 write: 10
P2 write: 11
P1 read:  (11)
```

**内存一致性**定义了对于**不同内存位置**的读写行为（被其他处理器观察到的行为）：
- 一致性只保证对地址 X 的写入最终会传播到其他处理器
- 一致性处理的是：相对于对其他地址的读写，对 X 的写入**何时**传播到其他处理器

**更直观的解释**：
- 缓存一致性的目标是使并行计算机的内存系统表现得**好像缓存不存在**——就像单处理器系统中缓存透明一样
- 没有缓存的系统就不需要缓存一致性
- 内存一致性定义了并行系统中对不同地址的 load/store 的**允许行为**——无论缓存是否存在，都需要指定内存的允许行为

### 2.4 TL;DR 总结

- 多处理器以**不直观和奇怪的方式**重排序内存操作
- 这种行为是**性能所必需的**
- **应用程序员**很少看到这种行为
- **系统（OS 和编译器）开发者**经常看到这种行为

## 3. 顺序一致性 (Sequential Consistency)

### 3.1 内存操作的四种排序

程序定义了一系列 load 和 store（即"程序顺序"）。

**四种内存操作排序类型：**

| 排序 | 含义 | 说明 |
|------|------|------|
| **W→R** | 对 X 的写入必须先于对 Y 的后续读取提交 | Write → Read |
| **R→R** | 对 X 的读取必须先于对 Y 的后续读取提交 | Read → Read |
| **R→W** | 对 X 的读取必须先于对 Y 的后续写入提交 | Read → Write |
| **W→W** | 对 X 的写入必须先于对 Y 的后续写入提交 | Write → Write |

**"写入必须在后续读取之前提交"的含义**：当写入在程序顺序中先于读取时，写入必须提交（其结果可见）到读取发生时。

### 3.2 多处理器执行示例

```c
// 初始: A = B = 0

// 处理器 0           // 处理器 1
(1) A = 1;           (3) B = 1;
(2) print B;         (4) print A;
```

**可能的输出是什么？**
- `"01"` — 可能：P0 先执行完，然后 P1 执行
- `"10"` — 可能：P1 先执行完，然后 P0 执行
- `"11"` — 可能：交错执行
- `"00"` — **不应该**发生（见下文分析）

### 3.3 不应该发生的排序

**程序不应该打印 `"00"` 或 `"10"`（对于特定分析）**。

使用 **happens-before 图**来推理：

```
要得到 r1=0, r2=0 (即 "00"):
  P0: A=1 → print B(0)  需要 (3) B=1 在 (2) 之后
  P1: B=1 → print A(0)  需要 (1) A=1 在 (4) 之后

如果图中存在环，则该结果不可能——某个事件必须在自身之前发生！
```

- `"00"` 输出意味着出现了循环依赖，在直觉的顺序一致性下不可能
- `"01"` 和 `"10"` 是合理的结果

### 3.4 顺序一致性的定义

**Lamport 1976（2013 图灵奖获得者）提出的顺序一致性：**

1. **所有操作按某个串行顺序执行**，如同它们在操作单个共享内存
2. **每个线程的操作按其程序顺序发生**

一个顺序一致的内存系统保持**全部四种**内存操作排序：
- W→R, R→R, R→W, W→W

```
所有内存操作的全局时序（包含地址 X 和 Y）:
P0 store: X ← 5
P1 store: X ← 10
P0 store: Y ← 1
P1 load:  X      // 读取哪个值取决于时序
P0 load:  X
P1 store: Y ← 20
```

### 3.5 顺序一致性：交换机隐喻

```
处理器 0 ──┐
处理器 1 ──┤
处理器 2 ──┼── [内存] (像交换机一样选择一个处理器执行)
处理器 3 ──┘

- 所有处理器按程序顺序发出 load 和 store
- 内存随机选择一个处理器，将该处理器的内存操作完整执行，
  然后再选择另一个处理器...
```

### 3.6 顺序一致性示例：逐步演示

```c
// 初始: A = 0, B = 0
处理器 0:        处理器 1:
  A = 1;           B = 1;
  r1 = B;          r2 = A;
```

**SC 下的执行步骤（"交换机"模型）：**

| 步骤 | 执行的操作 | 内存状态 | 说明 |
|------|-----------|---------|------|
| 1 | — | A=0, B=0 | 初始状态 |
| 2 | P0: A=1 | A=1, B=0 | 交换机选择 P0，执行第一条指令 |
| 3 | P0: r1=B | A=1, B=0 | 继续 P0，r1=0 |
| 4 | P1: B=1 | A=1, B=1 | 交换机切换到 P1 |
| 5 | P1: r2=A | A=1, B=1 | 继续 P1，r2=1 |
| **结果** | r1=0, r2=1 | — | **输出 "01"** |

在 SC 下，`r1=0, r2=0`（输出 `"00"`）不可能。

## 4. 宽松一致性模型 (Relaxed Consistency Models)

### 4.1 为什么需要放松排序要求？

**动机**：性能！特别是**隐藏内存延迟**——当内存操作独立时，将其与其他操作重叠执行。

在缓存一致的系统中，内存访问可能比单处理器中的简单读取要复杂得多（需要查找数据、发送失效消息等）。

```
     严格排序:                允许重排序:
     Write A                 Write A
        ↓ (等待)             Read B   ← 在写A完成前读B!
     Read B                  
     vs. 写A的延迟被隐藏
```

### 4.2 顺序一致性的问题

```c
处理器 0:
  A = 1;    // 写操作延迟很大（几百个周期）
  r1 = B;   // 这条指令与 A=1 不冲突
             // 不需要等待第一条指令完成！
处理器 1:
  B = 1;
  r2 = A;
```

在 SC 下，处理器 0 必须等待 `A=1` 完成才能执行 `r1=B`，即使这两条指令没有依赖关系。

### 4.3 优化：写缓冲区 (Write Buffer)

```
处理器 0                    内存          处理器 1
A = 1 ──→ [写缓冲区: A=1]  A=0, B=0  [写缓冲区] ←── B = 1
r1 = B    (处理器从自己的       (处理器从自己的
           写缓冲区读写)          写缓冲区读写)
```

使用写缓冲区后：

```c
// 初始: A = B = 0
处理器 0:        处理器 1:
(1) A = 1;      (3) B = 1;
(2) r1 = B;     (4) r2 = A;

// 问题: r1 = r2 = 0 可能吗？
// SC:  不可能 (No)
// 写缓冲区: 可能!
```

每个处理器从自己的写缓冲区读写，写操作可能尚未传播到内存或其他处理器的缓存。

**写缓冲区的性能优势**：
- 基本方案 (SC)：处理器一次发出一个内存操作，阻塞直到完成
- 放松 W→R 排序：写延迟几乎完全隐藏

```
没有写缓冲区:             有写缓冲区:
 处理器 ─→ 缓存            处理器 ─→ 写缓冲区 ─→ 缓存
 Reads/Writes 串行        Reads 可以绕过 Writes
```

### 4.4 写缓冲区：谁在乎？

- **每个现代处理器都使用它们**：Intel x86、ARM、RISC-V
- 需要**更弱的内存模型**
- **TSO (Total Store Order，全存储排序)**：x86 使用的模型，比 SC 稍难推理

### 4.5 TSO 和处理器一致性 (PC)

**放松 W→R 排序：允许读取移动到写入之前。**

**Total Store Ordering (TSO)**：
- 处理器 P 可以在其写操作被所有处理器看到之前读取 B
- 处理器可以将其自己的读取移到自己的写入之前
- 其他处理器的读取不能返回 A 的新值，直到对 A 的写入被所有处理器观察到

**Processor Consistency (PC)**：
- **任何**处理器可以在写入被所有处理器观察到之前读取 A 的新值

**TSO 和 PC 的共同点**：
- 只放松 **W→R** 排序约束
- **W→W 约束仍然存在**：同一线程的写入不会被重新排序（按程序顺序发生）

### 4.6 重要澄清

缓存一致性问题之所以存在，是因为硬件在多处理器缓存中复制数据，这些副本必须保持一致。

而**宽松内存一致性问题源于内存操作重排序的优化**。一致性问题与系统中是否存在缓存**无关**。

### 4.7 PSO：允许写操作被重排序

**Partial Store Ordering (PSO)**：进一步放松 **W→W** 排序。

```c
// 线程 1 (在 P1 上)       // 线程 2 (在 P2 上)
A = 1;                     while (flag == 0);
flag = 1;                  print A;
```

在 PSO 下，P2 可能观察到 `flag` 的变化先于 `A` 的变化——即使程序顺序中 `A=1` 在 `flag=1` 之前（写入被重排序了）。

**这意味着线程 2 可能打印出 `0`（A 的旧值）！** 尽管它看到 flag 已经被置为 1。

### 4.8 为什么需要更激进的重排序？

**W→W 重排序**：处理器可能在写缓冲区中重排序写操作（例如一个写操作缓存缺失而另一个命中）

**R→W 和 R→R 重排序**：处理器可能在指令流中重排序独立指令（乱序执行）

这些都是完全有效的优化，如果程序只有**单个指令流**的话。

### 4.9 允许所有重排序：弱排序 (WO) 和释放一致性 (RC)

**四种排序全部放松**：
| 排序 | 是否放松 |
|------|---------|
| W→R | 放松 |
| R→R | 放松 |
| R→W | 放松 |
| W→W | 放松 |

**对数据操作没有任何保证——一切都可以被重排序。**

动机是更高的性能：
- 在内存系统中重叠多个读取和写入
- 尽可能早地执行读取，尽可能晚地执行写入，以隐藏内存延迟

**示例模型**：
- **Weak Ordering (WO)**：弱排序
- **Release Consistency (RC)**：释放一致性

### 4.10 各一致性模型对比

| 模型 | W→R | R→R | R→W | W→W | 典型架构 |
|------|-----|-----|-----|-----|---------|
| **SC** (Sequential Consistency) | 保持 | 保持 | 保持 | 保持 | 理想模型 |
| **TSO** (Total Store Order) | 放松 | 保持 | 保持 | 保持 | x86/x64 |
| **PC** (Processor Consistency) | 放松 | 保持 | 保持 | 保持 | (理论) |
| **PSO** (Partial Store Order) | 放松 | 保持 | 保持 | 放松 | SPARC |
| **WO** (Weak Ordering) | 放松 | 放松 | 放松 | 放松 | (理论) |
| **RC** (Release Consistency) | 放松 | 放松 | 放松 | 放松 | ARM (类 RC) |

## 5. 同步机制与内存屏障

### 5.1 同步来救援

内存重排序看起来很糟糕（它确实是！），但每个架构都提供了**同步原语**使内存排序更严格。

**Fence（内存屏障）指令**防止重排序，但开销较大：
- 在屏障之前的所有内存操作必须在屏障之后的任何内存操作开始之前完成

```
  ...可重排序的读写...
  ═══════ MEMORY FENCE ═══════
  ...可重排序的读写...
  ═══════ MEMORY FENCE ═══════
  ...可重排序的读写...
```

**其他同步原语**（针对单个地址）：
- Read-Modify-Write / Compare-and-Swap (CAS)
- 事务内存 (Transactional Memory)

### 5.2 x86 和 ARM 的同步指令

**Intel x86/x64 (~ TSO)**：
- `mm_lfence`（"load fence"）：等待所有 load 完成
- `mm_sfence`（"store fence"）：等待所有 store 完成
- `mm_mfence`（"mem fence"）：等待所有内存操作完成

**ARM 处理器**：非常宽松的一致性模型，需要更仔细的屏障使用。

### 5.3 使用内存屏障保护共享数据

```c
// 使用屏障确保正确的发布/消费语义

// 线程 1: 发布数据
data = 42;           // 准备数据
mm_sfence();         // 存储屏障：确保 data 的写入先于 flag
flag = 1;            // 设置标志，通知数据就绪

// 线程 2: 消费数据
while (flag == 0);   // 等待标志
mm_lfence();         // 加载屏障：确保 flag 的读取先于 data
print(data);         // 保证读到 42
```

## 6. 数据竞争 (Data Races)

### 6.1 数据竞争的定义

一个**数据竞争 (data race)** 发生在以下条件同时满足时：
- 两个访问针对**同一内存位置**
- 至少一个是**写操作**
- 访问之间**没有同步操作排序**（unsynchronized）

### 6.2 冲突数据访问

两个不同处理器的内存访问**冲突**，如果：
- 它们访问同一内存位置
- 至少一个是写操作

**未同步的程序**：
- 冲突访问没有通过同步操作排序（如 fence、release/acquire 语义的操作、barrier 等）
- 未同步的程序包含数据竞争：程序输出取决于处理器的相对速度（**非确定性**程序结果）

**我们目前为止的所有示例都涉及数据竞争。**

### 6.3 同步程序与 "SC for DRF"

**同步程序在非 SC 系统上产生 SC 结果**：
- 同步程序是**数据竞争自由 (Data-Race-Free, DRF)** 的
- 如果没有数据竞争，重排序行为**不重要**
- 访问被同步操作排序，同步强制了顺序一致性

在实际中，大多数程序都是同步的（通过锁、barrier 等同步库实现），而不是通过临时的共享变量读写。

**"SC for DRF"** 范式：
- 如果程序没有数据竞争，程序员可以假定顺序一致性的行为
- 硬件和编译器负责处理底层的内存模型复杂性
- 这大大简化了并行编程

## 7. 语言级内存模型

### 7.1 为什么语言也需要内存模型？

**原因 1：编译器的优化不应对程序员可见**

不仅硬件会重排序内存操作，**编译器也会重排序**。如果编译器重排序了共享变量的内存操作，可能导致在 SC 硬件上也无法获得 SC 行为。

**原因 2：语言必须向程序员提供合约**

语言级内存模型告诉程序员编译器将如何（或不会如何）重排序他们的内存操作。例如：
- 编译器保证不重排序共享内存操作
- 这种优化约束对程序员是**可见的**

### 7.2 语言级内存模型的具体规定

现代语言（**C11, C++11**）和不那么现代的语言（**Java 5**）保证：

> **对数据竞争自由的程序 (DRF programs) 提供顺序一致性**
> 即 "SC for DRF"

- 编译器将插入必要的同步操作以适应硬件内存模型
- **如果程序包含数据竞争，则没有任何保证！**
- 直觉上：大多数程序员认为有竞争的程序是 buggy 的

**最佳实践：使用同步库！**

### 7.3 C++11 内存模型——原子操作与内存序

C++11 引入了 `std::atomic` 和内存序 (memory order) 的概念，允许程序员精细控制同步行为。

#### 7.3.1 基本原子操作

```cpp
#include <atomic>

std::atomic<int> flag(0);
std::atomic<int> data(0);

// 线程 1: 生产者
void producer() {
    data.store(42, std::memory_order_relaxed);   // 准备数据
    flag.store(1, std::memory_order_release);     // 发布：flag 之前的写入对消费者可见
}

// 线程 2: 消费者
void consumer() {
    while (flag.load(std::memory_order_acquire) == 0);  // 获取：之后的操作不会被重排到此之前
    int val = data.load(std::memory_order_relaxed);     // 保证看到 42
}
```

#### 7.3.2 C++ 内存序 (Memory Order)

| 内存序 | 含义 | 适用操作 |
|--------|------|---------|
| `memory_order_relaxed` | 无排序约束，只保证原子性。操作可能被任意重排序。 | 计数器递增等不需要同步的场景 |
| `memory_order_acquire` | 获取语义：此 load 之后的所有内存操作不会被重排到此 load 之前 | load 操作 |
| `memory_order_release` | 释放语义：此 store 之前的所有内存操作不会被重排到此 store 之后 | store 操作 |
| `memory_order_acq_rel` | 同时具有 acquire 和 release 语义 | read-modify-write 操作 |
| `memory_order_seq_cst` | 顺序一致性：全局统一的操作顺序。**默认选项** | 所有操作 |

#### 7.3.3 Release-Acquire 语义详解

Release-Acquire 是 C++ 内存模型中的核心同步机制：

```
线程 1 (Release):              线程 2 (Acquire):
  data = 42;           ─┐        while (flag == 0);
  flag.store(1, rel);   ┼── happens-before ──→  val = data;
                         │                      // 保证 val == 42
  所有在 release 之前的 │  所有在 acquire 之后的
  写入对 acquire 可见   └→ 操作不会被重排到 acquire 之前
```

**Release 保证**：
- 在 release store 之前的**所有内存写入**（包括非原子的）对后续 acquire 该变量的线程可见

**Acquire 保证**：
- 在 acquire load 之后的**所有内存操作**不会被重排序到此 acquire load 之前

**关键点**：Release-Acquire 对形成了一个 **happens-before** 关系。

#### 7.3.4 顺序一致性与 Acquire-Release 的对比

```cpp
// SC 版本 (默认): 更强的保证，但代价更高
std::atomic<int> x(0), y(0);

// 线程 1                     // 线程 2
x.store(1);                   y.store(1);
int r1 = y.load();            int r2 = x.load();

// 在 SC 下: r1 == 0 && r2 == 0 不可能
// SC 保证所有线程看到单一全局操作顺序
```

```cpp
// Acquire-Release 版本: 较弱但更高效
// 线程 1                          // 线程 2
x.store(1, memory_order_release);  y.store(1, memory_order_release);
int r1 = y.load(memory_order_acquire);  int r2 = x.load(memory_order_acquire);

// 在 Acquire-Release 下: r1 == 0 && r2 == 0 可能!
// 因为不同线程的独立 release/acquire 对之间没有全局排序
```

#### 7.3.5 完整示例：使用 C++ 原子实现锁

```cpp
#include <atomic>

class SpinLock {
    std::atomic<bool> locked{false};

public:
    void lock() {
        // 自旋直到成功获取锁
        while (locked.exchange(true, std::memory_order_acquire)) {
            // 自旋等待 (可以使用 relaxed load 优化)
            while (locked.load(std::memory_order_relaxed)) {
                // PAUSE 指令 (减少功耗)
            }
        }
        // acquire 语义: 临界区内的操作不会被重排到 lock 之前
    }

    void unlock() {
        // release 语义: 临界区内的操作在 unlock 之前对所有线程可见
        locked.store(false, std::memory_order_release);
    }
};
```

#### 7.3.6 Happens-Before 关系

在 C++ 内存模型中，**happens-before** 关系定义了操作之间的可见性保证：

1. **程序顺序 (Program Order)**：同一线程中，A 在 B 之前 → A happens-before B
2. **同步 (Synchronization)**：线程 T1 的 release 操作 synchronizes-with 线程 T2 的 acquire 操作（如果 T2 读取了 T1 写入的值）
3. **传递性 (Transitivity)**：如果 A happens-before B 且 B happens-before C，则 A happens-before C

```
线程 1:                  线程 2:
  data = 42;              |
  | (sequenced-before)    |
  flag.store(1, release)  |
           \              |
            \---- synchronizes-with
                          |
                          flag.load(acquire) == 1
                          | (sequenced-before)
                          val = data;  // 保证 val == 42

传递性: data=42 happens-before val=data
  （data=42 先于 release，release 同步于 acquire，acquire 先于 val=data）
```

#### 7.3.7 常见错误：使用 Relaxed 排序

```cpp
// 错误! 仅使用 relaxed 排序
std::atomic<int> data(0);
std::atomic<int> flag(0);

// 线程 1                     // 线程 2
data.store(42, relaxed);       while (flag.load(relaxed) == 0);
flag.store(1, relaxed);        int val = data.load(relaxed);
                               // val 不保证是 42!
                               // 硬件可能重排序 data 和 flag 的写入顺序
                               // 编译器也可能重排序
```

**正确做法**：
```cpp
// 使用 release-acquire 配对
data.store(42, memory_order_relaxed);     // 数据不需要同步
flag.store(1, memory_order_release);      // 只对 flag 使用 release

while (flag.load(memory_order_acquire) == 0);  // 对 flag 使用 acquire
int val = data.load(memory_order_relaxed);     // 保证看到 42
```

## 8. 总结

### 8.1 缓存一致性总结

- 缓存一致性问题存在是因为**单一共享地址空间的抽象不是由单一存储单元实现的**
- 存储分布在主存和本地处理器缓存之间，数据被复制用于提高性能
- **监听式 (snooping) 缓存一致性**：当可能影响一致性的缓存操作发生时，向所有其他缓存控制器广播通知
- 硬件架构师的挑战：最小化一致性实现的开销
- 软件开发者的挑战：注意由一致性协议引起的人为通信（如伪共享）
- 通过目录 (directory-based) 方法扩展缓存一致性

### 8.2 宽松一致性总结

- **动机**：通过允许内存操作重排序获得更高性能（顺序一致性不允许重排序）
- **代价**：软件复杂性——程序员或编译器必须正确插入同步以确保特定的操作排序
- 但实际上复杂性被封装在库中，提供了直观的原语（如 lock/unlock、barrier、fence）
- 优化常见情况：大多数内存访问不冲突，所以不要设计需要为此付出代价的系统
- 宽松一致性模型的区别在于它们忽略了哪些内存排序约束

### 8.3 内存一致性模型总结

| 概念 | 关键点 |
|------|--------|
| **内存一致性模型** | 定义硬件和编译器允许的内存操作重排序——是硬件/编译器与应用程序软件之间的**合约** |
| **弱模型** | 为提高性能而需要对操作进行更激进的重排序 |
| **SC for DRF** | 对数据竞争自由的程序保证顺序一致性，模型细节可以隐藏在同步库中 |
| **数据竞争** | 如果有数据竞争，程序行为是未定义的——使用同步库！ |
| **C++11 内存模型** | 提供 `std::atomic` 和多种 memory order 选项，支持 release-acquire 语义和 SC for DRF |

### 8.4 核心设计权衡

```
正确性 (Correctness)
    ↕
性能 (Performance)
    ↕
可编程性 (Programmability)

顺序一致性 (SC):  最易编程，性能最低
宽松模型 (RC/WO): 最难编程，性能最高
SC for DRF:       平衡——对大多数程序员提供 SC 的抽象，同时允许底层优化
```
