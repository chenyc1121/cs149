## 2. 缓存基础知识回顾

### 2.1 缓存示例回顾

**示例1：空间局部性与时间局部性**

假设：缓存总容量 8 字节，缓存行 (cache line) 大小为 4 字节（共 2 个缓存行），替换策略为 LRU。

访问序列：`0x0, 0x1, 0x2, 0x3, 0x0, 0x0, 0x4, 0x1`

```
时间线：
  0x0 → cold miss (加载 Line 0x0-0x3)
  0x1 → hit (同一缓存行内的空间局部性)
  0x2 → hit
  0x3 → hit
  0x0 → hit (时间局部性)
  0x0 → hit
  0x4 → cold miss (加载 Line 0x4-0x7，与 Line 0x0-0x3 共存)
  0x1 → hit
```

这里体现了两种数据局部性：

- **空间局部性 (Spatial Locality)**：加载一个缓存行会“预加载”同一行中其他地址的数据，后续对这些地址的访问会命中缓存
- **时间局部性 (Temporal Locality)**：对同一地址的重复访问会导致缓存命中

**示例2：容量缺失 (Capacity Miss)**

访问序列：`0x0, 0x1, 0x2, 0x3, 0x4, 0x5, 0x6, 0x7, 0x8, 0x9, 0xA, 0xB, 0xC, 0xD, 0xE, 0xF, 0x0`

```
时间线：
  0x0 → cold miss (加载 Line 0x0-0x3)
  0x1-0x3 → hit
  0x4 → cold miss (加载 Line 0x4-0x7)
  0x5-0x7 → hit
  0x8 → cold miss (加载 Line 0x8-0xB, 淘汰 Line 0x0-0x3)
  0x9-0xB → hit
  0xC → cold miss (加载 Line 0xC-0xF, 淘汰 Line 0x4-0x7)
  0xD-0xF → hit
  0x0 → capacity miss (加载 Line 0x0-0x3, 淘汰 Line 0x8-0xB)
```

**3C 缓存缺失模型 (3 Cs Cache Miss Model)**：

| 类型 | 英文名 | 描述 |
|------|--------|------|
| 冷缺失 | Cold (Compulsory) | 第一次访问某地址，必然缺失 |
| 容量缺失 | Capacity | 缓存容量不足以容纳所有需要访问的数据 |
| 冲突缺失 | Conflict | 由于缓存关联度有限，多个地址映射到同一缓存位置 |

### 2.2 缓存设计要点

**一个缓存行的典型结构（Intel 现代处理器：64 字节）：**

```
┌──────────────────┬────────────┬───────────┐
│ Data (64 bytes)  │    Tag     │ Line State│ Dirty Bit │
└──────────────────┴────────────┴───────────┘
```

对于语句 `int x = 1;`（假设 x 位于地址 0x12345604）：

- 处理器向地址执行写入操作
- 缓存控制器根据地址的 tag 部分查找对应缓存行
- 更新对应字节，并设置 dirty bit

**关键问题**：

- **写回 (Write-Back) vs 写穿透 (Write-Through)**：写回缓存仅在缓存行被淘汰时才写入主存；写穿透缓存每次写操作都立即写入主存
- **写分配 (Write-Allocate) vs 写不分配 (Write-No-Allocate)**：写分配在写缺失时先将缓存行加载到缓存再写入；写不分配则直接写入主存，绕过缓存

### 2.3 写分配 + 写回缓存的写缺失流程（单处理器）

处理器执行 `int x = 1;` 时，如果地址不在缓存中：

1. 处理器执行写操作，发生缓存缺失 (miss)
2. 缓存选择目标位置，如果该位置当前有脏行（dirty line），则先将脏行写回主存
3. 缓存从主存加载整个缓存行（分配缓存行）
4. 整个缓存行（64 字节）被加载，其中的 32 位（4 字节）被更新
5. 缓存行标记为脏 (dirty)

│4-way │       │4-way │       │4-way │       │4-way │

                  │              │

                  │              │

   │         Shared L3 Cache (2 MB per bank)     │

详细参数：

| 缓存层级 | 容量 | 关联度 | 策略 | 带宽 | 延迟 | 归属 |
|---------|------|--------|------|------|------|------|
| L1 Data | 32 KB | 8-way | Write-back | 2x32B load + 1x32B store/clock | ~4 cycles | 每核私有 |
| L2 | 256 KB | 4-way | Write-back | 64B/clock | ~12 cycles | 每核私有 |
| L3 (per bank) | 2 MB | 16-way | Inclusive | 32B/clock | ~42 cycles | 芯片共享 |

- **缓存行大小**：64 字节（所有层级统一）
- **支持能力**：72 个未完成的 load 操作，56 个未完成的 store 操作

## 4. 共享地址空间模型

### 4.1 抽象模型

在共享地址空间模型中：

- 线程通过读写共享变量来进行隐式通信
- 线程 1 存储值到 X，之后线程 2 读取 X（应该观察到线程 1 更新后的值）
- 锁等同步原语用于保证互斥访问

这是顺序编程的自然扩展。

### 4.2 共享内存多处理器的合理预期

```
Processor 0    Processor 1    Processor 2    Processor 3
    │               │               │               │
    └───────────────┼───────────────┼───────────────┘
                    │
              Interconnect
                    │
            ┌───────┴───────┐
            │    Memory     │
            │     I/O       │
            └───────────────┘
```

**直观预期**：读取地址 X 的值应该返回**任何处理器最后一次写入地址 X 的值**。

## 5. 缓存一致性问题 (The Cache Coherence Problem)

### 5.1 问题描述

现代处理器将内存内容复制到本地缓存中以提高性能，但这导致了一个严重问题：**不同处理器可能对同一内存位置观察到不同的值**。

```
Processor 0    Processor 1    Processor 2    Processor 3
   [Cache]       [Cache]        [Cache]        [Cache]
     │               │               │               │
     └───────────────┼───────────────┼───────────────┘
                     │
               Interconnect
                     │
                  Memory
```

### 5.2 具体示例

假设变量 `foo` 存储于地址 X，初始值为 0，缓存使用写回策略。

| 操作 | P1 $ | P2 $ | P3 $ | P4 $ | mem[X] | 说明 |
|------|------|------|------|------|--------|------|
| 初始状态 | - | - | - | - | 0 | foo=0 |
| P1 store X (写 1) | 1 | - | - | - | 0 | P1 缓存行变为 dirty |
| P1 load Y (淘汰 X) | - | - | - | - | 2 | X 被淘汰，但值 1 可能没写回？ |
| P3 load X | - | - | 0 | - | 0 | P3 从内存读取到旧值！ |
| P3 store X (写 2) | - | - | 2 | - | 0 | P3 写入新值 |
| P2 load X | - | 2 | 2 | - | 0 | P2 读到 P3 缓存的值 |
| P2 load X (hit) | - | 0 | 2 | - | 0 | P2 命中但值不对？ |
| P1 load X (miss) | 0 | 0 | 2 | - | 0 | P1 重新加载，得到旧值 |

**关键发现**：这不是互斥问题 (mutual exclusion problem)！即使加了锁也无法解决。这是由**多个本地缓存复制同一地址的数据**造成的硬件一致性问题。

## 6. 一致性的定义

### 6.1 “最后一次写入”的问题

“读取地址 X 应该返回最后一次写入 X 的值”这个直观定义存在模糊之处：

- “最后一次”是什么意思？两个处理器同时写入怎么办？
- 如果 P1 写入后，P2 紧接着读取，时间间隔太短以至于无法将写入信息传递给 P2 怎么办？

在顺序程序中，“最后一次”由**程序顺序** (program order) 决定，而非物理时间。在并行程序中，需要一个跨线程的顺序描述。

### 6.2 形式化定义：Coherence

内存系统是**一致的 (coherent)**，当且仅当：

对于每个内存位置，存在一个假想的、所有处理器对该位置操作的串行顺序，该顺序与程序执行结果一致，并且：

1. **单处理器顺序**：任何一个处理器发出的内存操作按其发出顺序执行
2. **写值可读性**：一次读操作返回的值是该位置在这个串行顺序中最后一次写入的值

```
地址 X 的操作时序：
  P0 write: 5
  P1 read (5)
  P2 read (5)
  P0 read (5)
  P1 write: 25
  P0 read (25)
```

### 6.3 缓存一致性不变量 (Cache Coherence Invariants)

对于任意内存地址 x，在任意时间段 (epoch) 内：

**单写者多读者不变量 (SWMR — Single-Writer, Multiple-Read)**：

- **读写时期 (Read-Write Epoch)**：仅有一个处理器可以写 x（也可以读 x）
- **只读时期 (Read-Only Epoch)**：可以有多个处理器只能读 x

**数据值不变量 (Data-Value Invariant / Write Serialization)**：

- 一个时期开始时内存地址的值 = 上一个读写时期结束时该内存地址的值

```
Address x:
  [Read-Write: P0] → [Read-Only: P0,P1,P2] → [Read-Write: P1] → [Read-Only: P0,P1] → ...
        ↓                      ↓                      ↓                      ↓
      epoch                   epoch                  epoch                  epoch
```

## 7. 实现一致性的方法

### 7.1 基于软件的方案（粗粒度：VM 页级别）

- 操作系统利用缺页 (page-fault) 机制来传播写入
- 可用于在集群工作站上实现内存一致性
- **主要问题**：伪共享 (false sharing) 导致严重性能问题

### 7.2 基于硬件的方案（细粒度：缓存行级别）

- **监听式一致性 (Snooping-based Coherence)**：本节课重点
- **目录式一致性 (Directory-based Coherence)**：简要介绍

### 7.3 共享缓存方案：最简单的一致性实现

```
Processor 0    Processor 1    Processor 2    Processor 3
    │               │               │               │
    └───────────────┼───────────────┼───────────────┘
                    │
              Interconnect
                    │
            ┌───────┴───────┐
            │  Shared Cache │  ← 所有处理器共享一个缓存
            └───────┬───────┘
                    │
                  Memory
```

**优点**：消除了多缓存复制状态的问题

**缺点**：
- 无法扩展（缓存的目的是本地且快速，共享违背了这一原则）
- 来自多个客户端的干扰/争用导致冲突缺失增加（破坏性）

但共享缓存也有好处：
- 便于细粒度共享（工作集重叠时）
- 一个处理器的 load/store 可能为另一个处理器预取缓存行（建设性）

例如，对于 `forall (i=0; i<N; i++) x[i] = y[i] + y[i+1] + y[i+2];`，如果多个处理器共享 L2/L3 缓存，一个核对 y 数组的加载可能让相邻核受益。

**Sun Niagara 2 (UltraSPARC T2) 案例**：
- 8 核共享 L2 缓存，通过 Crossbar Switch 连接
- 注意：Crossbar (CCX) 的面积大约等于一个核的面积

## 8. 监听式缓存一致性 (Snooping Cache Coherence)

### 8.1 核心思想

所有与一致性相关的活动都**广播**给系统中的所有处理器（更准确地说，广播给处理器的缓存控制器）。

缓存控制器“监听”(snoop) 内存操作，并按照缓存一致性协议来维护内存一致性。

```
Processor 0    Processor 1    Processor 2
  [Cache]        [Cache]        [Cache]
  Controller     Controller     Controller
    │               │               │
    └───────────────┼───────────────┘
                    │
          Bus / Interconnect
                    │
                 Memory
```

缓存控制器必须响应“两端”的操作：

1. **本地处理器端**：来自本地 CPU 的 load/store 请求
2. **总线/互联端**：来自互联上广播的一致性相关活动

### 8.2 最简单的实现：写穿透 + 无效化协议

**假设**：
1. 使用写穿透 (write-through) 缓存
2. 一致性粒度为缓存行

**协议**：
- 写操作时，缓存控制器广播**无效化消息 (invalidation message)**
- 其他缓存的同一行被无效化
- 下次其他处理器读取时，触发缓存缺失，从内存获取更新后的值（由于写穿透策略，内存已经是最新的）

```
操作序列示例：
  P1 load X      →  cache miss for X (从内存加载，值为 0)
  P0 load X      →  cache miss for X (从内存加载，值为 0)
  P0 write 100 to X →  invalidation for X (广播无效化), 内存写为 100
  P1 load X      →  cache miss for X (从内存加载，值为 100 ✓)
```

### 8.3 写穿透的局限性

**问题**：每次写操作都要访问主存，带宽需求极高。

**写回缓存**可以吸收大部分写流量（写命中时），显著降低带宽需求。但这也意味着需要更复杂的一致性协议来维护缓存一致性不变量。

## 9. 写回缓存中的缓存一致性

### 9.1 核心挑战

使用写回缓存后，内存中的数据可能已经“过时”(stale)，因为最新的值可能只在某个处理器的 dirty 缓存行中。

```
P0: Write X
      ↓
  [P0 Cache: X=dirty] ── 内存中有 X 的旧值
      ↓
P1: Read X
      ↓
  如果 P1 直接从内存加载 X → 得到旧值！不一致！
```

### 9.2 Dirty 状态的新含义

在写回缓存的一致性协议中，缓存行的 **dirty 状态**现在表示**独占所有权 (Exclusive Ownership)**：

- **Modified (已修改)**：该缓存是唯一拥有该行有效副本的缓存（可以安全写入）
- **Owner (所有者)**：该缓存负责在其它处理器尝试加载时传播数据（否则其它处理器会从内存获得旧数据）

**总线的两个重要性质**（支持监听式一致性的基础）：
1. **广播性**：总线上的所有事务对所有缓存控制器可见
2. **串行化**：总线事务按顺序执行，提供全局顺序

## 10. MSI 写回无效化协议

### 10.1 协议概述

缓存一致性协议是维护缓存一致性不变量的算法。每个处理器的缓存控制器在以下两种情况下执行该协议：

- 本地处理器的 load 和 store 操作
- 总线上来自其他缓存的消息

所有缓存控制器“协作”来维护一致性。

### 10.2 MSI 三种缓存行状态

| 状态 | 含义 | 说明 |
|------|------|------|
| **M (Modified)** | 已修改 / 独占 | 缓存行仅在一个缓存中有效且已修改（dirty），内存中的数据是旧的。等同于单处理器中的 dirty 状态 + 独占含义。 |
| **S (Shared)** | 共享 | 缓存行在一个或多个缓存中有效，内存中的数据是最新的。 |
| **I (Invalid)** | 无效 | 缓存行无效，等同于单处理器缓存中的 invalid。 |

### 10.3 操作类型

**处理器操作（由本地 CPU 触发）**：

| 缩写 | 含义 | 说明 |
|------|------|------|
| PrRd | Processor Read | 处理器读 |
| PrWr | Processor Write | 处理器写 |

**总线事务（来自远程缓存）**：

| 缩写 | 含义 | 说明 |
|------|------|------|
| BusRd | Bus Read | 获取缓存行副本，无修改意图 |
| BusRdX | Bus Read Exclusive | 获取缓存行副本，有修改意图（排他性读取） |
| BusWB | Bus Writeback | 将脏行写回内存 |

### 10.4 MSI 状态转换图

```
                     PrRd / --
                ┌──────────────────┐
                │                  │
                ▼                  │
            ┌───────┐    PrWr /    │
            │       │  BusRdX      │
            │   M   │◄─────────────│
            │(Modif)│              │
            └───┬───┘              │
    BusRdX /   │                  │
    BusWB      │ PrRd / --        │
                ▼                  │
            ┌───────┐              │
            │       │  PrWr /      │
            │   S   │  BusRdX      │
            │(Shared)──────────────│
            └───┬───┘              │
    BusRd /    │                  │
    BusWB      │                  │
                ▼                  │
            ┌───────┐              │
            │   I   │◄─────────────┘
            │(Inval)│  BusRdX / --
            └───────┘  PrWr / --
                       BusRd / --
```

**图例说明**：
- `A / B`：表示如果缓存控制器观察到事件 A，则执行动作 B
- **实线箭头**：处理器发起 (Processor initiated)
- **虚线箭头**：总线发起 (Bus initiated)

### 10.5 状态转换详解

#### 从 I (Invalid) 状态出发：

| 触发事件 | 执行动作 | 新状态 | 说明 |
|----------|----------|--------|------|
| PrRd | BusRd | S | 读缺失：在总线上发出 BusRd 请求，获取共享副本 |
| PrWr | BusRdX | M | 写缺失：在总线上发出 BusRdX 请求，获取独占副本 |

#### 从 S (Shared) 状态出发：

| 触发事件 | 执行动作 | 新状态 | 说明 |
|----------|----------|--------|------|
| PrRd | -- | S | 读命中：无需总线事务 |
| PrWr | BusRdX | M | 写命中但不独占：发出 BusRdX 升级为 M（Upgrade） |
| BusRd | -- | S | 其它缓存读取同一行，保持共享 |
| BusRdX | -- | I | 其它缓存需独占写，本缓存无效化该行 |

#### 从 M (Modified) 状态出发：

| 触发事件 | 执行动作 | 新状态 | 说明 |
|----------|----------|--------|------|
| PrRd | -- | M | 读命中：无需总线事务，可以继续安全读写 |
| PrWr | -- | M | 写命中：无需总线事务，独占状态下可直接写入 |
| BusRd | BusWB | S | 其它缓存读取：将脏数据写回内存，降级为共享 |
| BusRdX | BusWB | I | 其它缓存需独占写：将脏数据写回内存，自己无效化 |

### 10.6 协议核心逻辑总结

1. **读操作获取“共享”状态**：即使只有这一个缓存有副本（数据来自内存）
2. **写操作前必须获得独占所有权**：BusRdX 导致其它缓存无效化该行
3. **如果在 S 状态写命中**：仍然需要发出 BusRdX——称为**升级 (upgrade)**事务（因为 S 状态不意味着独占，必须通知其他缓存）
4. **M 状态意味着独占**：可以直接读写，无需总线事务
5. **如果另一个缓存的 BusRdX 命中了本缓存的 M 行**：本缓存必须先写回数据 (BusWB)，然后无效化该行

### 10.7 MSI 协议示例

单写者、多读者协议的执行示例：

| 步骤 | 操作 | P1 状态 | P2 状态 | P3 状态 | 总线事务 | 数据来自 |
|------|------|---------|---------|---------|----------|----------|
| 1 | P1 read x | **S** | -- | -- | BusRd | Memory |
| 2 | P3 read x | S | -- | **S** | BusRd | Memory |
| 3 | P3 write x | I | -- | **M** | BusRdX | Memory |
| 4 | P1 read x | **S** | -- | S→? | BusRd | **P3 Cache** |
| 5 | P1 read x (hit) | S | -- | S | -- | P1 Cache |
| 6 | P2 write x | I | **M** | I | BusRdX | Memory |

**关键观察**：
- 步骤 4：P1 读 x 时，P3 拥有 M 状态的缓存行，P3 向 P1 提供数据（而非从内存获取旧数据）
- P3 在提供数据的同时将状态从 M 降级到 S（因为数据不再独占）
- 步骤 5：P1 再次读 x 时，缓存命中，无需总线事务

### 10.8 MSI 如何满足缓存一致性

**1. SWMR 不变量**：

- 同一时刻只有一个缓存可以处于 M 状态（其他缓存在收到 BusRdX 时会无效化）
- 多个缓存可以同时处于 S 状态（只读）

**2. 数据值不变量（写串行化）**：

- 当收到 BusRd 或 BusRdX 时，处于 M 状态的缓存提供数据（而非从内存获取旧值）
- 总线对所有事务进行串行化，提供了全局顺序

## 11. MESI 协议

### 11.1 MSI 协议的低效之处

在 MSI 协议中，即使应用程序完全没有共享数据，也存在效率问题：

**常见场景：先读后写 — 需要两次总线事务！**

```
  1. PrRd: BusRd    → I → S (第一次总线事务)
  2. PrWr: BusRdX   → S → M (第二次总线事务，升级)
```

对于“读取一个地址，然后修改它”这种极为常见的编程模式，MSI 需要两次互联事务。

### 11.2 MESI 新增状态：E (Exclusive Clean)

**解决方案**：增加一个新的状态 **E (Exclusive Clean / 独占干净)**

- 缓存行未被修改（与 S 相同，内存中的数据是有效的）
- 但只有这个缓存拥有该行的副本（与 M 相同，是独占的）
- 将**独占性**与**行所有权**解耦：E 状态下行不脏，内存副本有效
- **优势**：从 E 升级到 M 不需要总线事务（因为已知自己是唯一持有者）

### 11.3 MESI 四种状态

| 状态 | 含义 | 是否独占 | 是否脏 | 内存是否最新 |
|------|------|----------|--------|------------|
| **M** (Modified) | 已修改 | 是 | 是 | 否 |
| **E** (Exclusive) | 独占干净 | 是 | 否 | 是 |
| **S** (Shared) | 共享 | 否 | 否 | 是 |
| **I** (Invalid) | 无效 | - | - | - |

### 11.4 MESI 状态转换图

```
                       PrRd / --
                  ┌──────────────────┐
                  │                  │
                  ▼                  │
              ┌───────┐              │
              │   M   │              │
              │(Modif)│              │
              └───┬───┘              │
   BusRdX/BusWB │                   │
   BusRd/BusWB  │                   │
                │                   │
                ▼                   │
              ┌───────┐  PrWr / --  │
              │   E   │◄────────┐   │
              │(Exclu)│         │   │
              └───┬───┘         │   │
    BusRdX / --  │              │   │
    BusRd / --   │              │   │
       (other    │              │   │
        asserts  │              │   │
        shared)  ▼              │   │
              ┌───────┐         │   │
              │   S   │────┐    │   │
              │(Shared)│    │    │   │
              └───┬───┘    │    │   │
    BusRd / --   │         │    │   │
    BusRdX / --  │         │    │   │
                 ▼         │    │   │
              ┌───────┐    │    │   │
              │   I   │    │    │   │
              │(Inval)│    │    │   │
              └───────┘    │    │   │
                  ▲        │    │   │
                  │        │    │   │
                  └────────┴────┴───┘
            PrWr/BusRdX  PrWr/BusRdX

  关键转换路径：
    I → E : PrRd + BusRd (没有其他缓存声明共享：claimed 'no share')
    I → S : PrRd + BusRd (有其他缓存声明共享)
    I → M : PrWr + BusRdX
    E → M : PrWr (无需总线事务！仅需本地操作)
    E → S : BusRd (其他缓存读取，降级为共享)
    E → I : BusRdX (其他缓存请求排他访问)
    S → M : PrWr + BusRdX (需要升级事务)
    M → S : BusRd + BusWB (其他缓存读取，写回后降级)
    M → I : BusRdX + BusWB (其他缓存请求排他，写回后无效化)
```

**关键转换说明**：

当一个缓存行的 `PrRd` 触发 `BusRd` 时：
- 如果**没有**其他缓存声明也拥有该行（无 shared 信号），则该行进入 **E 状态**（独占但不脏）
- 如果**有**其他缓存声明也拥有该行（有 shared 信号），则该行进入 **S 状态**（共享）

### 11.5 MESI vs MSI 的效率对比

常见模式 "读后修改" 的总线事务数量：

| 协议 | 第一次读 | 随后修改 | 总线事务总数 |
|------|---------|---------|------------|
| MSI | BusRd (I→S) | BusRdX (S→M) | **2 次** |
| MESI | BusRd (I→E) | 无需总线 (E→M) | **1 次** |

**MOESI 协议**（MESI 的进一步扩展）：增加 **O (Owned)** 状态，允许共享脏数据。MESI+O 在 AMD 处理器中广泛使用。

## 12. 目录式缓存一致性 (Directory-Based Coherence)

### 12.1 监听式方案的局限性

- 监听式方案通过**广播**一致性消息来确定其他缓存中缓存行的状态
- **不可扩展**：随着处理器数量增加，广播带宽成为瓶颈

### 12.2 目录式方案的核心思想

**避免广播**：将缓存行状态信息集中存储在一个地方——**目录 (Directory)**。

```
      Processor + Cache     Processor + Cache
            │                      │
            │   point-to-point     │
            └──────────┬───────────┘
                       │
               ┌───────┴───────┐
               │   Directory   │  ← 记录每个缓存行在所有缓存中的状态
               └───────┬───────┘
                       │
                    Memory
```

- 目录中每个缓存行条目包含该行在所有缓存中的状态信息
- 缓存根据需要从目录查询信息
- 缓存一致性通过缓存之间的**点对点消息**（而非广播）在“按需通知”的基础上维护
- 仍然需要维护两个不变量：SWMR 和写串行化

### 12.3 Intel Core i7 中的目录一致性

```
Core 0    ── L1 / L2 ──┐
Core 1    ── L1 / L2 ──┤
Core 2    ── L1 / L2 ──┼── Shared L3 Cache (目录) ── Ring Interconnect
Core 3    ── L1 / L2 ──┘
```

- **L3 作为集中式目录**：为 L3 中所有缓存行维护目录信息
- L3 是**包含式缓存 (inclusive cache)**：L2 中的任意行一定也存在于 L3
- 目录维护包含该行的 L2 缓存列表
- 一致性消息只发送给包含该行的 L2 缓存，而非广播给所有 L2
- 目录维度：P=4（4 个核心），M=L3 缓存行数

**延迟对比（Core i7 Xeon 5500 系列）**：

| 访问类型 | 大约延迟 |
|----------|---------|
| L1 hit | ~4 cycles |
| L2 hit | ~10 cycles |
| L3 hit (行未共享) | ~40 cycles |
| L3 hit (行在另一个核心中共享) | ~65 cycles |
| L3 hit (行在另一个核心中被修改) | ~75 cycles |
| 本地 DRAM | ~30 ns (~120 cycles) |
| 远程 DRAM (NUMA) | ~100 ns (~400 cycles) |

## 13. 对程序员的影响：通信开销

### 13.1 多处理器中增加的访问延迟

通信时间是并行计算的关键开销之一。在多处理器系统中，表现为内存访问时间增加：

```
                          Uniprocessor          Multiprocessor
  Register          ── 寄存器，更少的寄存器分配
  L1 Cache          ── L1 缓存，更高的缺失率
  L2 Cache          ── L2 缓存，更高的缺失率
  Main Memory       ── 主存，在 NUMA 中可能“缺失”
  Remote            ── 远程访问，额外长时间延迟
```

**平均内存访问时间 (AMAT)**：

```
AMAT = Σ (frequency of accessᵢ × latency of accessᵢ)

AMAT_Multiprocessor > AMAT_Uniprocessor
```

即使只有一小部分的额外延迟也可能对性能产生显著影响。

## 14. 伪共享 (False Sharing)

### 14.1 问题描述

**伪共享**：两个处理器写入不同的地址，但这些地址映射到同一个缓存行。

- 缓存行在两个写处理器的缓存之间**乒乓**传递
- 由于一致性协议，产生了大量的通信开销
- 这完全是**人为的通信 (artifactual communication)**——本质上不需要通信（因为两个处理器访问的是不同变量），但因为它们在同一缓存行上而被强制通信

```
  P1: 频繁写 &counter[0]   ┐
                           ├── 同一缓存行！ ── 乒乓效应
  P2: 频繁写 &counter[1]   ┘
```

### 14.2 代码示例：伪共享问题

```c
// ===== 有伪共享问题的版本 =====
// 多个线程同时更新相邻的计数器，它们位于同一缓存行
void* worker(void* arg) {
    volatile int* counter = (int*)arg;
    for (int i = 0; i < MANY_ITERATIONS; i++)
        (*counter)++;
    return NULL;
}

void test1(int num_threads) {
    pthread_t threads[MAX_THREADS];
    int counter[MAX_THREADS];  // 所有计数器在连续的数组元素中
                                // 多个计数器位于同一缓存行！
    for (int i = 0; i < num_threads; i++)
        pthread_create(&threads[i], NULL, &worker, &counter[i]);
    for (int i = 0; i < num_threads; i++)
        pthread_join(threads[i], NULL);
}

// 执行时间 (8线程, 4核系统): 14.2秒  ← 非常慢！
```

```c
// ===== 消除伪共享的版本 =====
// 通过填充 (padding) 确保每个计数器独占一个缓存行
#define CACHE_LINE_SIZE 64

struct padded_t {
    int counter;
    char padding[CACHE_LINE_SIZE - sizeof(int)];  // 填充至一个缓存行大小
};

void test2(int num_threads) {
    pthread_t threads[MAX_THREADS];
    padded_t counter[MAX_THREADS];

    for (int i = 0; i < num_threads; i++)
        pthread_create(&threads[i], NULL, &worker, &(counter[i].counter));
    for (int i = 0; i < num_threads; i++)
        pthread_join(threads[i], NULL);
}

// 执行时间 (8线程, 4核系统): 4.7秒  ← 快了约3倍！
```

### 14.3 性能差异分析

| 版本 | 执行时间 | 原因 |
|------|---------|------|
| 无 padding | 14.2 秒 | 多个线程的计数器在同一缓存行，缓存行乒乓传递，产生大量一致性通信 |
| 有 padding | 4.7 秒 | 每个计数器独占一个缓存行，线程间无伪共享，几乎无额外通信 |

### 14.4 缓存行大小对缺失率的影响

从实际应用（Barnes-Hut, Radiosity, Ocean Sim, Radix Sort）的模拟结果来看：

- 随着缓存行大小增加（8B → 256B）：
  - **冷缺失 (Cold)** 和 **容量/冲突缺失 (Capacity/Conflict)** 通常减少（更好的空间局部性）
  - **真共享缺失 (True Sharing)** 和 **伪共享缺失 (False Sharing)** 会增加（更大的缓存行意味着更多的无关数据被打包到同一行）
  - **升级缺失 (Upgrade)** 也可能增加

- 64 字节的缓存行大小在现代系统中是一个较好的平衡点

## 15. 总结：缓存一致性

### 15.1 核心要点

1. **缓存一致性问题**之所以存在，是因为“单一共享地址空间”这一抽象并非由单一物理存储单元实现：
   - 存储分布在主存和本地处理器缓存之间
   - 数据在本地缓存中被复制以提高性能

2. **监听式缓存一致性**的核心思想：
   - 每当发生可能影响一致性的缓存操作时，缓存控制器向系统中所有其他缓存控制器**广播通知**
   - 所有缓存控制器独立执行一致性协议逻辑，协作维护一致性

3. **不变量** (Invariants)：
   - SWMR (Single-Writer Multiple-Read)：任何一个时刻，要么一个写者，要么多个读者
   - 数据值不变量/写串行化：新 epoch 开始时的值 = 上一读写 epoch 结束时的值

4. **协议演进**：
   - **MSI**：3 状态 - Modified, Shared, Invalid
   - **MESI**：4 状态 - 增加 Exclusive Clean，减少常见场景下的总线事务
   - **MOESI**：5 状态 - 增加 Owned，支持共享脏数据

5. **可扩展性**：
   - 监听式方案受限于广播一致性消息的能力
   - 目录式方案通过点对点消息和集中式目录来扩展一致性

### 15.2 硬件架构师的挑战

- 最小化一致性实现的开销
- 在保持正确性的前提下降低通信量

### 15.3 软件开发者的注意事项

- 警惕由一致性协议引起的**人为通信**（如伪共享）
- 在多线程编程中，尽可能让每个线程的数据在独立的缓存行上
- 可以使用 padding 技术或编译器属性（如 `alignas(64)` 或 `__cacheline_aligned`）来避免伪共享
- 使用性能分析工具（如 Intel VTune）来分析内存系统的性能表现
