## 1. 第一课回顾：程序与处理器基础

### 1.1 什么是计算机程序？

程序本质上是一系列处理器指令的列表。高级语言编写的代码经过编译器编译后，转化为处理器能够执行的机器指令。

**C 代码示例**：

```c
int main(int argc, char** argv) {
  int x = 1;
  for (int i=0; i<10; i++) {
    x = x + x;
  }
  printf("%d\n", x);
  return 0;
}
```

编译后生成汇编指令，每条指令对应一个简单的操作（如 `pushq`、`movq`、`subq`、`addl`、`jmp` 等）。程序的执行流程由 PC（程序计数器）控制，逐条读取并执行指令。

### 1.2 处理器如何执行指令？

一个简化的处理器模型包含以下核心组件：

- **寄存器（Registers）**：维护程序状态，存储变量的值，作为运算的输入和输出
- **执行单元（ALU / Execution Unit）**：执行指令所描述的操作，可能修改寄存器或内存中的值
- **取指/译码（Fetch/Decode）**：确定下一条要执行的指令

**Kayvon 的极简处理器模型**：

```
取指/译码  -->  执行上下文（寄存器 R0-R3）  -->  ALU（执行单元）
```

该处理器每个时钟周期执行一条指令，工作流程为：
1. Fetch/Decode 取指并译码
2. 从寄存器或内存读取操作数
3. ALU 执行计算
4. 将结果写回寄存器或内存

**示例指令序列**：

```asm
ld   r0, addr[r1]     ; 从内存加载数据到 r0
mul  r1, r0, r0       ; r1 = r0 * r0
mul  r1, r1, r0       ; r1 = r1 * r0
...
st   addr[r2], r0      ; 将 r0 存储到内存
```

## 2. 指令级并行与超标量处理器

### 2.1 什么是指令级并行（ILP）？

指令级并行（Instruction Level Parallelism）是指在一个指令序列中，某些指令之间没有数据依赖关系，因此它们可以同时执行。

**示例程序**：

```c
a = 2
b = 4
tmp2 = a + b        // 6
tmp3 = tmp2 + a     // 8
tmp4 = b + b        // 8
tmp5 = b * b        // 16
tmp6 = tmp2 + tmp4  // 14
tmp7 = tmp5 + tmp6  // 30
if (tmp3 > 7)
  print tmp3
else
  print tmp7
```

指令依赖图展示了指令之间的数据依赖关系。例如：
- `tmp3 = tmp2 + a` 依赖于 `tmp2 = a + b`
- `tmp4 = b + b` 与 `tmp2` 无关，可以并行计算
- `tmp6 = tmp2 + tmp4` 依赖于前面两条指令

### 2.2 超标量处理器（Superscalar Processor）

超标量处理器能够在一个时钟周期内**译码并执行多条指令**（前提是这些指令在指令流中是独立的）。

**超标量处理器的核心特性**：

- 具有多个取指/译码单元和多个执行单元
- 通过**乱序控制逻辑（Out-of-order Control Logic）** 自动发现指令流中的独立指令，并在多个执行单元上并行执行
- **"尊重程序顺序"** 的含义：超标量处理器虽然可以乱序执行，但最终结果必须与顺序执行一致（即必须正确处理数据依赖）

```
取指/译码1 --> 执行单元1
取指/译码2 --> 执行单元2
        乱序控制逻辑
```

**关键点**：超标量执行是硬件自动完成的，程序员无需显式指定并行性。但可用的 ILP 受限于指令流中实际存在的独立指令数量。

## 3. 内存系统与缓存层次结构

### 3.1 程序的线性内存地址空间

计算机内存被组织为一个**字节数组**，每个字节由其"地址"标识（假设内存是按字节寻址的）。

```
地址    值
0x0     16
0x1     255
0x2     14
0x3     0
0x4     128
...
0x10    6
...
0x1F    0
```

在这个示例中，程序的地址空间大小为 32 字节（有效地址范围 0x0 到 0x1F）。

### 3.2 Load 指令：访问内存

```asm
ld R0 ← mem[R2]
```

这条指令的含义是："请从寄存器 R2 中存储的地址开始，加载一个四字节的值到寄存器 R0 中。"

**示例**：

```
R0: 96
R1: 64
R2: 0xff681080      <-- 存储目标地址
R3: 0x80486412

内存:
0xff681080: 42      <-- 将 42 加载到 R0
0xff681084: 32
0xff681088: 0
0xff68107c: 1024
```

### 3.3 内存访问延迟与停顿（Stalls）

**内存访问延迟（Memory Access Latency）**：指内存系统向处理器提供数据所需的时间。例如：100 个时钟周期，或 100 纳秒。

**停顿（Stall）**：当处理器无法执行指令流中的下一条指令时，就发生了停顿——原因是后续指令依赖于尚未完成的前序指令。

内存访问是停顿的主要来源：

```asm
ld r0 mem[r2]    ; 加载操作（需要等待内存）
ld r1 mem[r3]    ; 加载操作（需要等待内存）
add r0, r0, r1   ; 依赖上面的两个加载结果，必须等待
```

内存访问时间约为数百个时钟周期。"访问时间"是延迟的度量指标。

## 4. 缓存详解：局部性与替换策略

### 4.1 什么是缓存？

缓存是对内存抽象的一种硬件实现细节，**不影响程序的输出，只影响程序的性能**。

**缓存的核心特性**：

- 缓存是**片上存储**，保存内存中部分数值的副本
- 如果某个地址的数据"在缓存中"，处理器从该地址加载/存储数据的速度远快于从 DRAM 直接访问
- 缓存以**缓存行（Cache Line）** 为粒度操作

**缓存示例**（容量 2 行，每行 4 字节）：

```
数据缓存:
行地址    行内值
0x4       [0, 0, 6, 0]
0xC       [255, 0, 0, 0]
```

### 4.2 缓存替换策略

处理器如何决定缓存中保存哪些数据？

- **直接映射缓存（Direct Mapped Cache）**
- **组相联缓存（Set-Associative Cache）**
- **缓存行（Cache Line）**

简化理解：容量为 N 字节的缓存存储最近访问的 N 个地址的数据。

**LRU（Least Recently Used，最近最少使用）替换策略**：为给新数据腾出空间，丢弃缓存中访问时间距今最久的数据。

### 4.3 缓存示例 1：空间局部性与时间局部性

**假设条件**：
- 总缓存容量：8 字节
- 每行 4 字节（共 2 行）
- LRU 替换策略
- 内存为 16 字节数组

**访问序列**：`0x0, 0x1, 0x2, 0x3, 0x0, 0x4, 0x1, 0x0, 0x4, 0x1`

- 首次访问 `0x0` → **冷缺失（Cold Miss）**，加载行 0x0
- 访问 `0x1, 0x2, 0x3` → **命中（Hit）**，数据已在缓存行中
- 访问 `0x0` → 命中
- 访问 `0x4` → **冷缺失**，加载行 0x4

**两种数据局部性**：

| 类型 | 说明 |
|------|------|
| **空间局部性（Spatial Locality）** | 加载一个缓存行时"预加载"了同一行中后续访问需要的数据，导致缓存命中 |
| **时间局部性（Temporal Locality）** | 对同一地址的重复访问导致命中 |

### 4.4 缓存示例 2：容量缺失

**访问序列**：遍历整个 16 字节数组，然后再次遍历

```
地址访问: 0x0 → 0x1 → 0x2 → ... → 0xF → 0x0 → 0x1 → ...
```

- 访问 `0x0-0x3`：行 0x0 被加载
- 访问 `0x4-0x7`：行 0x4 被加载
- 访问 `0x8-0xB`：行 0x8 被加载（**驱逐行 0x0**）
- 访问 `0xC-0xF`：行 0xC 被加载（**驱逐行 0x4**）
- 再次访问 `0x0`：**容量缺失（Capacity Miss）**，需要重新加载行 0x0（驱逐行 0x8）

**讨论问题**：为什么第二次读取地址 0x0 时没有命中？因为缓存容量不够，行 0x0 被之前加载行 0x8 时驱逐了。

### 4.5 缓存层次结构

现代计算机中线性内存地址空间抽象的**实际实现非常复杂**。

```
处理器 <--> L1 Cache (32 KB)  <--> L2 Cache (256 KB)
       <--> L3 Cache (20 MB)  <--> DRAM (64 GB)
```

**缓存层次结构的组织原则**：

| 层级 | 容量 | 延迟特征 |
|------|------|----------|
| L1 Cache | 32 KB | 最靠近处理器，延迟最低 |
| L2 Cache | 256 KB | 中等容量和延迟 |
| L3 Cache | 20 MB | 更大容量，更高延迟 |
| DRAM（主存） | 64 GB | 最大容量，最高延迟 |

### 4.6 数据访问延迟数字（Kaby Lake CPU，4 GHz）

| 数据位置 | 延迟（时钟周期数） |
|----------|-------------------|
| L1 Cache | 4 cycles |
| L2 Cache | 12 cycles |
| L3 Cache | 38 cycles |
| DRAM（最佳情况） | ~248 cycles |

### 4.7 数据移动的高能耗成本

现代系统设计中的经验法则：**始终寻求减少计算机中的数据移动量**。

**估算数字**：

| 操作 | 能耗 |
|------|------|
| 整数运算 | ~1 pJ |
| 浮点运算 | ~20 pJ |
| 从片上 SRAM 读取 64 位（1mm 距离） | ~26 pJ |
| 从移动低功耗 DRAM（LPDDR）读取 64 位 | ~1200 pJ |

**含义**：
- 以 10 GB/sec 从内存读取数据：约 1.6 瓦
- 移动 GPU 的整个功率预算：约 1 瓦
- iPhone 6 电池：约 7 瓦时
- MacBook Pro 电池：约 99 瓦时

**结论**：利用局部性（locality）至关重要！

## 5. 多核处理器架构

### 5.1 从单核到多核的时代转变

**多核时代之前的处理器设计**：

```
取指/译码 --> 执行上下文 --> 执行单元（ALU）
           大数据缓存
           乱序控制逻辑
           复杂分支预测器
           内存预取器
```

芯片上绝大多数晶体管用于帮助**单条指令流运行得更快**。更多晶体管 = 更大的缓存、更智能的乱序逻辑、更好的分支预测器。

**多核时代的设计理念（Idea #1）**：

与其使用更多晶体管来增加加速单条指令流的处理器逻辑复杂度（如乱序和推测执行），不如使用不断增加的晶体管数量为处理器**增加更多核心**。

### 5.2 多核并行执行

**双核处理器并行执行**：

```
核心1: 取指/译码 --> 执行上下文 --> ALU
核心2: 取指/译码 --> 执行上下文 --> ALU
```

- 每个核心可能比原来的"高级"核心慢（例如慢 25%）
- 但有两个核心：2 x 0.75 = 1.5（有加速潜力）

**关键问题**：如果程序本身没有表达并行性，多核处理器反而会比原来的"高级"单核更慢。

**本节的计算示例程序**（泰勒级数计算 sin(x)）：

```c
void sinx(int N, int terms, float* x, float* y)
{
   for (int i=0; i<N; i++)
   {
    float value = x[i];
    float numer = x[i] * x[i] * x[i];
    int denom = 6;  // 3!
    int sign = -1;
    for (int j=1; j<=terms; j++)
    {
       value += sign * numer / denom;
       numer *= x[i] * x[i];
       denom *= (2*j+2) * (2*j+3);
       sign *= -1;
      }
      y[i] = value;
   }
}
```

该程序使用泰勒展开计算 sin(x)：sin(x) = x - x^3/3! + x^5/5! - x^7/7! + ...

## 6. 用代码表达并行性

### 6.1 使用 C++ 线程表达并行

```c
typedef struct {
   int N;
   int terms;
   float* x;
   float* y;
} my_args;

void my_thread_func(my_args* args)
{
   sinx(args->N, args->terms, args->x, args->y); // do work
}

void parallel_sinx(int N, int terms, float* x, float* y)
{
   std::thread my_thread;
   my_args args;
   args.N = N/2;
   args.terms = terms;
   args.x = x;
   args.y = y;
   my_thread = std::thread(my_thread_func, &args);  // 启动线程
   sinx(N - args.N, terms, x + args.N, y + args.N); // 主线程工作
   my_thread.join();  // 等待线程完成
}
```

### 6.2 数据并行表达式（forall 构造）

在 Kayvon 设想的编程语言中使用 `forall` 结构声明循环迭代是独立的：

```c
void sinx(int N, int terms, float* x, float* y)
{
   // 声明循环迭代是独立的
   forall (int i from 0 to N)
   {
    float value = x[i];
    float numer = x[i] * x[i] * x[i];
    int denom = 6;  // 3!
    int sign = -1;
    for (int j=1; j<=terms; j++)
    {
       value += sign * numer / denom;
       numer *= x[i] * x[i];
       denom *= (2*j+2) * (2*j+3);
       sign *= -1;
      }
      y[i] = value;
   }
}
```

### 6.3 数据并行的关键特性

这段代码的一个重要特性：**并行性存在于循环的各次迭代之间**。所有迭代执行的是**完全相同的指令序列**（由循环体定义），但作用于**不同的输入数据**（由 x[i] 给出）。

利用 `forall` 声明的程序可以：
- 编译器自动生成多核并行代码
- 编译器自动生成向量指令以利用核心内的 SIMD 处理能力

## 7. SIMD 处理：单指令多数据

### 7.1 SIMD 的核心理念（Idea #2）

**SIMD = Single Instruction, Multiple Data（单指令多数据）**

将管理指令流的成本/复杂度**分摊到多个 ALU 上**。

```
取指/译码 --> 执行上下文
          ALU0 ALU1 ALU2 ALU3 ALU4 ALU5 ALU6 ALU7
          (相同的指令广播到所有 ALU，并行执行)
```

同一条指令被广播到所有 ALU，操作在全部 ALU 上并行执行。

### 7.2 使用 AVX 向量内置函数的程序

原来的标量程序编译后一次处理一个数组元素（标量指令，标量寄存器，32 位浮点数）。

使用 AVX 内置函数改写后的向量程序：

```c
#include <immintrin.h>

void sinx(int N, int terms, float* x, float* y)
{
   float three_fact = 6;  // 3!
   for (int i=0; i<N; i+=8)
   {
      __m256 origx = _mm256_load_ps(&x[i]);
      __m256 value = origx;
      __m256 numer = _mm256_mul_ps(origx, _mm256_mul_ps(origx, origx));
      __m256 denom = _mm256_broadcast_ss(&three_fact);
      int sign = -1;
      for (int j=1; j<=terms; j++)
      {
         // value += sign * numer / denom
         __m256 tmp = _mm256_div_ps(_mm256_mul_ps(_mm256_set1ps(sign), numer), denom);
         value = _mm256_add_ps(value, tmp);
         numer = _mm256_mul_ps(numer, _mm256_mul_ps(origx, origx));
         denom = _mm256_mul_ps(denom, _mm256_broadcast_ss((2*j+2) * (2*j+3)));
         sign *= -1;
      }
      _mm256_store_ps(&y[i], value);
   }
}
```

**关键变化**：

| 特性 | 标量程序 | 向量程序 |
|------|---------|---------|
| 数据类型 | `float` | `__m256`（256 位向量） |
| 循环步长 | `i++` | `i+=8` |
| 加载指令 | 标量 `ld` | `_mm256_load_ps`（同时加载 8 个 float） |
| 运算指令 | 标量 `mul` | `_mm256_mul_ps`（同时计算 8 个 float） |
| 寄存器 | 32 位标量寄存器 | 256 位向量寄存器 |

编译后的向量程序使用**向量指令**一次处理 8 个数组元素：

```asm
vloadps  xmm0, addr[r1]     ; 加载 8 个 float
vmulps   xmm1, xmm0, xmm0   ; 8 个 float 同时相乘
vmulps   xmm1, xmm1, xmm0
...
vstoreps addr[xmm2], xmm0   ; 存储 8 个 float
```

### 7.3 16 个 SIMD 核心：128 个元素并行

扩展后的能力：
- 16 个核心
- 每个核心 8 个 SIMD ALU
- 总共 128 个 ALU
- 16 个同时执行的指令流
- 最多同时处理 128 个数据元素

## 8. 条件执行与分支发散

### 8.1 SIMD 中的条件执行问题

在 SIMD 模型中，当遇到条件分支（if-else）时，不同数据元素可能走不同的分支路径。

```c
forall (int i from 0 to N) {
    float t = x[i];
    if (t > 0.0) {
        t = t * 50.0;      // 执行 if 分支
        t = t + 100.0;
    } else {
        t = t * t;         // 执行 else 分支
        t = t + 30.0;
    }
    t = t / 10.0;          // 非条件代码
    y[i] = t;
}
```

### 8.2 使用掩码处理分支

SIMD 处理器的处理方式：

```
时钟周期 -- ALU1..ALU8 执行情况

  1      float t = x[i]              (所有 ALU 活跃)
  2      if (t > 0.0) 求值            (所有 ALU 活跃)
         结果: T T T F F F F F
  3      t = t * 50.0                (只有 ALU1-3 做有用工作)
  4      t = t + 100.0               (只有 ALU1-3 做有用工作)
  5      t = t * t                   (只有 ALU4-8 做有用工作，条件为 false)
  6      t = t + 30.0                (只有 ALU4-8 做有用工作)
  7      t = t / 10.0                (所有 ALU 恢复活跃)
  8      y[i] = t                    (所有 ALU 活跃)
```

**关键观察**：并非所有 ALU 都做有用的工作！最坏情况下，在 8 宽 SIMD 处理器上可能只有 1/8 的峰值性能。

### 8.3 最坏情况思考题

**问题**：能否只用一条 `if` 语句构造出在 8 宽 SIMD 处理器上性能最差的代码？

**答案思路**：构造一个 if-else 分支，使得每次只有一个数据元素走 if 分支，其余 7 个走 else 分支。这样在 if 分支执行期间（可能非常长），ALU 利用率仅为 1/8。

### 8.4 重要术语

| 术语 | 含义 |
|------|------|
| **指令流一致性（Coherent Execution）** | 同一指令序列适用于多个数据元素的程序特性；SIMD 高效使用**必需**的条件 |
| **发散执行（Divergent Execution）** | 程序中缺乏指令流一致性的情况；不适用于 SIMD 高效执行，但不会妨碍不同核心之间的并行化 |

### 8.5 CPU 上的 SIMD 执行

现代 CPU 的 SIMD 指令集示例：

| 指令集 | 位宽 | 操作能力 |
|--------|------|---------|
| Intel AVX2 | 256 bit | 8 x 32 位 或 4 x 64 位 |
| Intel AVX-512 | 512 bit | 16 x 32 位 |
| ARM Neon | 128 bit | 4 x 32 位 |

**"显式 SIMD"**：SIMD 并行化在编译时完成。可以在程序二进制中看到 SIMD 指令（如 `vstoreps`、`vmulps` 等）。

SIMD 指令的生成方式：
1. 程序员使用内置函数（intrinsics）显式请求
2. 使用并行语言语义传达（如 `forall` 示例）
3. "自动向量化"编译器通过依赖分析推断

### 8.6 GPU 上的 SIMD 执行

**"隐式 SIMD"**：

- 编译器生成标量指令的二进制文件
- 但 N 个程序实例始终一起在处理器上运行
- **硬件**（而非编译器）负责同时执行来自多个程序实例的**相同指令**，作用于 SIMD ALU 上的不同数据

大多数现代 GPU 的 SIMD 宽度范围：**8 到 32**。发散执行可能是一个大问题——编写糟糕的代码可能以机器峰值能力的 1/32 运行。

## 9. 硬件多线程：隐藏内存延迟

### 9.1 多线程的核心理念（Idea #3）

**问题**：如果数据最近没有被读取（不在缓存中），且下一条要读取的数据不容易预测，怎么办？

```c
int x = some_function();
int y = A[x];  // A[x] 的值难以预测
```

**解决方案——硬件多线程**：

在同一个核心上交替处理多个线程，以隐藏停顿。如果当前线程无法继续推进……就去处理另一个线程。

### 9.2 多线程隐藏停顿的工作机制

**单线程核心**（1 个线程）：

```
时间 -->

Thread 1 (元素 0-7):
ALU0..ALU7 工作 --> 停顿(等待内存) --> ALU0..ALU7 工作 --> ...
```

**4 线程核心**：

```
时间 -->

Thread 1 (元素 0-7):  可运行 --> 停顿 --> 可运行 --> 停顿 --> 完成
Thread 2 (元素 8-15):           可运行 --> 停顿 --> 可运行 --> 停顿 --> ...
Thread 3 (元素 16-23):                    可运行 --> 停顿 --> 可运行 --> ...
Thread 4 (元素 24-31):                               可运行 --> 停顿 --> ...
```

当 Thread 1 停顿时，核心切换到 Thread 2、3、4 执行，保持 ALU 持续忙碌。

### 9.3 吞吐量计算：一个权衡

**吞吐量导向系统的核心理念**：

潜在地**增加任何单个线程完成工作的总时间**，以便在运行多个线程时**提高整体系统吞吐量**。

注意：一个线程可能处于"可运行但不被执行"的状态（核心正在执行来自另一个线程的指令）。

### 9.4 存储执行上下文的成本

执行上下文的片上存储是**有限资源**：

| 策略 | 隐藏延迟能力 | 每线程工作集大小 |
|------|------------|----------------|
| 16 个硬件线程（每线程小工作集） | **高** | 小 |
| 4 个硬件线程（每线程大工作集） | **低** | 大 |

需要在延迟隐藏能力和每线程可用存储空间之间进行权衡。

### 9.5 计算核心利用率

**假设**：程序执行 3 条算术指令 + 1 次内存加载（12 周期延迟）。

**单线程情况**：

```
Thread 0: [3 arith] [--- 12 cycle stall ---] [3 arith] [--- 12 cycle stall ---] ...
利用率: 3/15 = 20%
```

**双线程情况**：

```
Thread 0: [3 arith] [------ stall ------] [3 arith] [------ stall ------]
Thread 1:           [3 arith] [------ stall ------] [3 arith] ...
利用率: 6/15 = 40%
```

**100% 利用率需要 5 个线程**（每条指令 = 1 个时钟周期）。

### 9.6 算术密集度对线程需求的影响

**修改假设**：程序执行 6 条算术指令 + 1 次内存加载（12 周期延迟）。

**此时 100% 利用率只需要 3 个线程**（算术与内存延迟的比例更高，因此更少的线程就能隐藏内存停顿）。

### 9.7 两项关键结论

**结论一**：具有多个硬件线程的处理器能够通过在其他线程上执行指令来避免停顿，当某个线程必须等待长延迟操作完成时。

注意：多线程**不改变**内存操作的延迟，只是**不再导致处理器利用率下降**。

**结论二**：多线程处理器通过执行其他线程的算术运算来隐藏内存延迟。**每个内存访问对应的算术运算越多，隐藏内存停顿所需的线程数越少。**

### 9.8 硬件多线程的两种形式

核心管理多个线程的执行上下文，但**ALU 资源数量不变**——多线程仅帮助在面对高延迟操作时更高效地使用它们。处理器每个时钟周期决定执行哪个线程。

| 形式 | 描述 |
|------|------|
| **交错多线程（Interleaved Multi-threading / Temporal Multi-threading）** | 每个时钟周期，核心选择一个线程，在核心的 ALU 上从该线程运行一条指令 |
| **同时多线程（Simultaneous Multi-threading, SMT）** | 每个时钟周期，核心从多个线程中选择指令在 ALU 上运行。例如：Intel 超线程（2 线程/核心） |

**V100 全芯片**：80 SM x 2048 = **163,840 个数据需要同时处理**以达到最大延迟隐藏能力。

## 11. 三种并行执行形式总结

| 形式 | 描述 | 并行性来源 |
|------|------|----------|
| **超标量（Superscalar）** | 在同一指令流中开发 ILP。在同一核心内并行处理同一指令流中的不同指令 | 硬件在执行时自动发现并行性 |
| **SIMD** | 多个 ALU 由同一指令控制（在同一核心内）。对数据并行工作负载高效——将控制成本分摊到多个 ALU | 编译器完成向量化（显式 SIMD）或运行时由硬件完成（隐式 SIMD） |
| **多核（Multi-core）** | 使用多个处理核心，提供线程级并行：在每个核心上同时执行完全不同的指令流 | 软件通过线程 API 创建线程以向硬件暴露并行性 |

### 关键图示：三种架构对比

```
超标量处理器（单核）:        多核处理器:                SIMD 四核处理器:
取指/译码 --> 执行1          取指/译码 --> ALU         取指/译码 --> 8宽SIMD ALU
取指/译码 --> 执行2          取指/译码 --> ALU         取指/译码 --> 8宽SIMD ALU
                                                     取指/译码 --> 8宽SIMD ALU
每时钟最多2条指令           每核每时钟1条指令           取指/译码 --> 8宽SIMD ALU
(来自同一指令流)            (来自不同指令流)           每核每时钟1条8宽SIMD指令
```

## 12. 高效利用现代并行处理器的条件

为了使现代并行处理器高效运行，应用程序必须满足：

1. **拥有足够多的并行工作**：以利用所有可用的执行单元（跨多个核心和每个核心内的多个执行单元）
2. **并行工作项组必须需要相同的指令序列**：以利用 SIMD 执行
3. **暴露比处理器 ALU 更多的并行工作**：以实现工作交错来隐藏内存停顿

## 14. 复习：从简单处理器到完整并行架构的演变

以下是本讲座内容的完整演进路径，展示了各种并行技术如何叠加组合：

### 14.1 阶段 1：简单单核处理器

```
取指/译码 --> 执行上下文(1个硬件线程) --> ALU --> 数据缓存 --> 内存
寄存器: R0-R7, PC
指令流: ld ... / mul ... / add ... / st ...
能力: 每个时钟周期 1 条标量指令
```

### 14.2 阶段 2：超标量核心

```
取指/译码1 --> [指令选择] --> ALU1
取指/译码2 --> [指令选择] --> ALU2
执行上下文(1个硬件线程), 寄存器 R0-R7, PC
```

**能力**：双路超标量——每条时钟周期从同一指令流中最多运行 2 条独立标量指令（来自同一硬件线程）。

### 14.3 阶段 3：SIMD 执行能力

```
取指/译码 --> 执行上下文 --> 8宽向量ALU (8个ALU并行)
寄存器: V0-V7 (向量寄存器), PC
指令流: vector_ld / vector_mul / vector_add / vector_st
```

**能力**：单个核心，单线程——每个时钟周期从一条指令流中运行 1 条 8 宽 SIMD 向量指令。

### 14.4 阶段 4：异构超标量（标量 + SIMD）

```
取指/译码1 --> [指令选择] --> 标量ALU
取指/译码2 --> [指令选择] --> 8宽向量ALU
执行上下文(1个硬件线程): 标量寄存器 R0-R3 + 向量寄存器 V0-V3
```

**能力**：双路超标量——每时钟周期最多 2 条独立指令（一条标量 + 一条向量），来自同一指令流。

### 14.5 阶段 5：多线程核心

```
取指/译码 --> 执行上下文0 + 执行上下文1 --> ALU --> 缓存 --> 内存
指令流0 + 指令流1
```

**能力**：多线程核心（2 个线程）——每个时钟周期从两个硬件线程之一运行 1 条标量指令。线程可以运行完全不同的指令流（处于不同位置），在时间上交错执行。

### 14.6 阶段 6：多线程 + 超标量核心

```
取指/译码1 --> [指令选择] --> 标量ALU
取指/译码2 --> [指令选择] --> 8宽向量ALU
执行上下文0 + 执行上下文1 (各有标量和向量寄存器)
指令流0 + 指令流1
```

**能力**：多线程双路超标量核心——每时钟周期最多 2 条独立指令（标量+向量），可来自同一线程或不同线程。支持交错执行和同时执行。

### 14.7 阶段 7：多核 + 多线程 + 超标量

```
核心0: 4线程 + 双路超标量(标量+向量) --> 共享数据缓存 --> 内存
核心1: 4线程 + 双路超标量(标量+向量) --> 共享数据缓存
指令流 0-7 (每个核心 4 个线程)
```

**能力**：双核处理器，每个核心 4 个线程，双路超标量。总共 8 个并发指令流。

### 14.8 真实案例：Intel Skylake/Kaby Lake 核心

- 双路多线程（2 线程/核心）
- 每核心可同时运行最多 4 条标量指令 + 最多 3 条 8 宽向量指令
- 包含多个功能单元：标量 ALU、8 宽向量 ALU（MUL/ADD）、L1/L2 缓存

### 14.9 GPU "SIMT" 模型

- 许多现代 GPU 执行仅包含**标量指令**的硬件线程
- GPU 核心检测不同硬件线程何时执行**相同指令**，并使用 SIMD ALU **同时执行**最多 SIMD 宽度的线程
- 执行不同指令的线程被**"掩码屏蔽"**（masked off）

## 15. 思考实验

**问题**：编写一个生成 2 个线程的应用程序，运行在如下处理器上：

```
双核处理器
每核心 2 个执行上下文
每时钟最多 2 条指令（其中 1 条可以是 8 宽 SIMD 指令）
```

- **Q**: "谁"负责将应用程序的线程映射到处理器的线程执行上下文？
- **A**: **操作系统（OS）**

- **Q**: 如果你正在实现 OS，你会如何将 2 个线程分配到 4 个执行上下文？
- **思路**：可以将 2 个线程分配到**不同核心**上（每个核心 1 个线程），最大化单线程性能（更多缓存和资源可用）；也可以分配到**同一核心**的 2 个上下文上，利用多线程隐藏延迟。

## 17. 补充：数据预取（Prefetching）

许多现代 CPU 具有**猜测未来数据访问**并提前将数据"预取"到缓存中的逻辑。

```
预估 r2 的值，提前发起加载 --> 数据到达缓存
预估 r3 的值，提前发起加载 --> 数据到达缓存
...
ld r0 mem[r2]   --> 缓存命中！
ld r1 mem[r3]   --> 缓存命中！
add r0, r0, r1
```

- 动态分析程序的内存访问模式以做出预测
- 预取减少停顿，因为数据在被访问时已在缓存中
- **缺点**：如果猜测错误，预取可能**降低性能**（消耗带宽、污染缓存）

## 18. 总结

Lecture 2 从软件工程师的视角讲解了计算机架构的核心概念，重点是现代并行处理器如何实现高吞吐量：

1. **超标量执行**：在同一核心内利用指令级并行（ILP），硬件自动发现独立指令并并行执行
2. **SIMD 执行**：单指令多数据，将控制成本分摊到多个 ALU，对数据并行工作负载高效
3. **多核执行**：多个处理核心同时执行不同的指令流，通过软件线程暴露并行性
4. **硬件多线程**：通过交错执行多个线程来隐藏内存访问延迟，提高处理器利用率
5. **内存层次结构**：L1/L2/L3 缓存 + DRAM，利用时间和空间局部性降低数据访问延迟
6. **缓存一致性**：对 SIMD 效率至关重要；分支发散会严重降低 SIMD 性能

理解这些基础知识有助于：
- 理解和优化并行程序的性能
- 获得关于哪些工作负载可能从快速并行机器中受益的直觉
