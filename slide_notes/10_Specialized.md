## 1. 引言：能量受限的计算时代

### 1.1 能量约束无处不在

现代计算系统普遍面临**能量受限（Energy-Constrained）**的根本性挑战：

| 场景 | 能量受限的原因 |
|------|---------------|
| 移动设备 | 有限的电池容量；无风扇散热限制 |
| 超级计算机 | 数十万 CPU/GPU 的庞大规模；数据中心供电与冷却成本 |
| 数据中心 | AI 需求的指数级增长；电力与冷却成为硬约束 |

### 1.2 AI 受限于能量

AI 模型的计算需求呈指数级增长，而数据中心面临严重的能量受限。这意味着单纯的性能提升不再是唯一目标——**能效比（Performance/Watt）**成为核心指标。

### 1.3 性能与功耗的基本关系

$$
\text{Power} = \frac{\text{Ops}}{\text{second}} \times \frac{\text{Joules}}{\text{Op}}
$$

提高能效的路径：
- **专用化（Specialization）** → 固定功能硬件 → 显著提升能效
- 更少的指令控制开销 → 更多的能量用于实际计算

## 2. 为什么通用处理器如此“低效”

### 2.1 执行一条指令的真实代价

在现代处理器上执行一条指令的完整流程：

```
1. 读取指令 (Instruction Fetch)
2. 解码指令 (Decode)
3. 检查依赖/流水线冲突 (Hazard Detection)
4. 识别可用的执行资源 (Resource Allocation)
5. 使用解码后的操作数控制寄存器文件 SRAM（读取数据）
6. 将数据从寄存器文件移动到选定的执行单元
7. 执行算术运算
8. 将结果从执行单元移回寄存器文件
9. 使用解码后的操作数控制写回寄存器文件

附加开销：
- 指令翻译为微指令 (uop)、访问 uop 缓存
- 地址翻译、与指令缓存通信
```

**关键洞察**：大部分能量消耗在**指令流水线的控制逻辑**上，而非实际的计算本身。

> 思考题：SIMD 执行如何降低某些类型计算的开销？这些计算需要具备什么特性？

### 2.2 H.264 视频编码的能量分布案例

以 H.264 视频编码为例，即使使用了 SIMD 指令，能量消耗分布如下：

| 组件 | 说明 |
|------|------|
| **FU** (Functional Units) | 功能单元 — 实际执行计算的部件 |
| **RF** (Register Fetch) | 寄存器读取 |
| **Ctrl** (Control) | 流水线控制逻辑 |
| **Pip** (Pipeline Registers) | 流水线级间寄存器 |
| **IF** (Instruction Fetch) | 指令获取 + 指令缓存 |
| **D-$** (Data Cache) | 数据缓存 |

编码的五个主要阶段：
1. 整像素运动估计 (integer motion estimation)
2. 分数亚像素运动估计 (fractional/subpixel motion estimation)
3. 帧内预测、DCT、量化 (intra-frame prediction, DCT, quantization)
4. 算术编码 (arithmetic encoding)

**结论**：即使使用 SIMD 优化实现，功能单元消耗的能量仅占总能量的一小部分。大部分能量被指令控制、寄存器访问和数据移动消耗。

> 参考：Hameed et al. ISCA 2010

## 3. 专用化的效率收益量化分析

### 3.1 FFT 基准测试：专用化优势

以快速傅里叶变换 (FFT) 为例（Chung et al. MICRO 2010）：

| 平台 | 面积效率 | 能效 |
|------|---------|------|
| **ASIC** | 与 CPU 相同性能仅需 **~1/1000** 芯片面积 | 相同性能仅需 **~1/100** 功耗 |
| **GPU** | 比 CPU 核心高效 **5-7 倍**（面积） | — |
| **FPGA** | 介于 GPU 与 ASIC 之间 | 介于 GPU 与 ASIC 之间 |

### 3.2 专用化的效率经验法则

> 对比基线：CPU 上高质量的 C 代码实现

| 架构类型 | 能效提升 | 适用条件 |
|----------|---------|---------|
| **吞吐量最大化处理器**（如 GPU） | 约 **10倍** perf/watt | 代码适合宽数据并行执行 + 计算受限 |
| **固定功能 ASIC** | 约 **100-1000倍** 甚至更高 | 计算受限 + 非纯浮点运算 |

### 3.3 指令控制开销的比例

通过更复杂的指令来分摊指令流的控制开销：

| 指令类型 | 控制开销占比 |
|----------|------------|
| 半精度 FMA（融合乘加） | ~2000% 开销 |
| 半精度 DP4（vec4 点积） | ~500% 开销 |
| 半精度 4x4 MMA（矩阵乘加） | **~27%** 开销 |

**关键原则**：用一条复杂指令执行多个操作，摊销指令流处理的开销。

## 4. 数字信号处理器 (DSP)

### 4.1 DSP 的设计哲学

DSP 是**可编程处理器**，但具有更简单的指令流控制路径：
- **复杂指令**（SIMD/VLIW）：每条指令执行多个操作，摊销控制成本
- 牺牲部分通用性换取更高的能效

### 4.2 实例：Qualcomm Hexagon DSP

- 用于 Qualcomm Snapdragon SoC 上的调制解调器、音频和图像处理
- 基于 **VLIW**（超长指令字）架构

**VLIW vs. SIMD 的区别**：
- **SIMD**：一条指令对多个数据执行**相同**的操作
- **VLIW**：一条指令指定多个**不同类型**的操作同时执行

```
Hexagon DSP 性能特征：
- FFT 最内层循环中，每个周期执行 29 个 RISC 等价操作
- 应用于 Google Pixel 手机
```

### 4.3 Anton 超级计算机：分子动力学专用

**Anton 1 (2008)**：模拟蛋白质的时间演化
- **512 个 ASIC** 用于计算粒子间相互作用
- 吞吐量优化的 FFT 子系统
- **自定义低延迟互连网络**，专为 N-body 模拟的通信模式设计
- 由 DE Shaw Research 开发

**Anton 3 (2025)**：比同期 GPU 快约 **20 倍**

## 5. FPGA：可编程逻辑的中间地带

### 5.1 FPGA 的核心概念

FPGA（现场可编程门阵列）位于 ASIC 和通用处理器之间：

| 特性 | ASIC | FPGA | 处理器 |
|------|------|------|--------|
| 可编程性 | 无 | 可重配置 | 完全可编程 |
| 能效 | 最高 | 中等-高 | 最低 |
| 开发成本 | $1000万-1亿+ | 中等 | 低 |

FPGA 架构组成：
- **可编程查找表 (LUT)**：实现组合逻辑
- **触发器 (Flip-Flop)**：存储状态（寄存器）
- **可编程互连**：连接各逻辑块

### 5.2 LUT 的工作原理

以 Xilinx Virtex-7 FPGA 的 **LUT6**（6 输入、1 输出）为例：

```
LUT6 = 64 元素表格（2^6 = 64 种输入组合）

示例：6-输入 AND 门 → 对应特定的 64 位真值表
40-输入 AND 门 → 通过级联 8 个 LUT6 的输出实现（延迟 = 3 级）
```

### 5.3 现代 FPGA 的组成

现代 FPGA 不仅包含可编程逻辑，还包含大量**硬核**：

- **存储器块 (SRAM)**：片上存储
- **DSP 块**：硬核乘法器
- **CPU 核心**（ARM、RISC-V）：提供通用计算能力

编程方式：使用硬件描述语言（如 **Verilog**）

### 5.4 云端 FPGA

- Amazon EC2 **F1/F2 实例**：可通过云服务使用 FPGA 加速

## 6. 效率与可编程性的权衡

### 6.1 能效-可编程性谱系

```
可编程性
   高 ▲
      │  ● 节能优化 CPU (easiest to program)
      │
      │  ● 吞吐量优化处理器 (GPU)  ~10x 更高效
      │
      │  ● 领域专用加速器 (DSL)     ~20x 更高效
      │        (Google TPU, 有限领域可编程性)
      │
      │  ● 可编程 DSP              ~20x 
      │
      │  ● FPGA/可重构逻辑         ~50x (尚在发展中)
      │        编程困难（降低使用门槛是活跃的研究方向）
      │
      ▼  ● ASIC                    100-1000x 更高效
   低          不可编程 + 设计/验证/制造成本数千万至数亿美元
              (视频编解码、音频播放、相机RAW处理、神经网络)
              
                    能效 →
```

### 6.2 ASIC 的适用场景

ASIC 适合以下**固定功能**场景：
- 视频编解码 (Video encode/decode)
- 音频播放 (Audio playback)
- 相机 RAW 处理 (Camera RAW processing)
- 神经网络推理（未来趋势）

## 7. AI 模型加速：GPU 的演进与局限

### 7.1 AI 模型为何适合 GPU

高性能 AI 模型实现通常以 GPU 为目标：
- **高算术强度**：计算特征类似于稠密矩阵-矩阵乘法
- 受益于 GPU 丰富的浮点运算能力
- 存在高度优化的内核库（如 **cuDNN**）

### 7.2 GPU 作为 AI 加速器的局限性

**核心问题**：GPU 仍然是通用处理器，是否需要如此强的通用性？

思考方向：
- 通用处理器的大量芯片面积用于**通用性**支持
- 如果目标仅为 AI/DNN 推理，可以去除这些通用性开销

### 7.3 理想 AI 模型加速器的特征

| 特征 | 原因 |
|------|------|
| **高分块张量**（Tiled Tensors, 如 16x16, 32x32） | 在 GEMM 上达到峰值 TFLOPS，低指令开销 |
| **异步计算**（Asynchronous Compute） | 重叠计算和内存访问 |
| **异步内存访问**（Asynchronous Memory Access） | 重叠计算和内存访问 |
| **异步芯片间通信** | 重叠计算、内存和通信 |
| **计算单元间通信** | 支持融合和流水线 |
| **流式数据流**（Streaming Dataflow） | —— |

### 7.4 AI 模型本质上是数据流图

AI 推理/训练的计算图天生就是**数据流图**：

```
   Weights         Sample
      │              │
      ▼              ▼
   GEMM 1 ──────►  Pool
                      │
                      ▼
   Weights ────►  GEMM 2
                      │
                      ▼
                   SoftMax
                      │
                      ▼
                     Sum
```

这种数据流特性暗示：**理想的底层硬件应当直接支持数据流执行模式**。

## 8. 理想 AI 加速器的特征

### 8.1 GEMM 的关键洞察

> **GEMM 计算本身很廉价，但数据移动非常昂贵。**
>
> 数据移动的成本体现在：硅面积、功耗（瓦特）、延迟（纳秒）

### 8.2 理想的 AI 加速器编程模型

- **CUTLASS**：NVIDIA 的 GPU 矩阵乘法模板库
- **Triton**：OpenAI 的 GPU 内核编程语言
- **Thunderkittens**：Stanford 的嵌入式 DSL

### 8.3 最小化数据移动成本

| 特征 | 目的 |
|------|------|
| 分块张量（16x16, 32x32） | 最大化 GEMM 的 TFLOPS |
| 低指令开销 | 将更多面积/能量用于计算 |
| 异步计算 | 重叠计算与内存访问 |
| 异步内存访问 | 隐藏内存延迟 |
| 异步芯片间通信 | 多芯片扩展时的效率 |
| 计算单元间通信 | 支持**融合 (Fusion)** 和**流水线 (Pipelining)** |
| 流式数据流 | 芯片上空间化布局计算 |

### 8.4 数据移动的能量成本

| 操作 | 能耗（数量级） |
|------|---------------|
| 整数运算 | ~1 pJ |
| 浮点运算 | ~20 pJ |
| 从 1mm 外小 SRAM 读取 64 位 | ~26 pJ |
| 从 LPDDR（移动 DRAM）读取 64 位 | ~1200 pJ |

> 规则：在现代系统设计中，始终寻求减少数据移动量。
>
> 来源：Bill Dally (NVIDIA), Tom Olson (ARM)

## 9. GPU AI 加速深入剖析：从 A100 到 B100

### 9.1 NVIDIA 芯片的专用化历程

NVIDIA GPU 正在变得越来越专用化：

| 芯片 | 新增硬件特性 |
|------|-------------|
| **V100** | 第一代 Tensor Core |
| **A100** | 第三代 Tensor Core, Tensor Core 稀疏性, 异步拷贝, L2 缓存驻留控制 |
| **H100** | 第四代 Tensor Core, FP8 数据格式, Transformer Engine, 异步执行, 分布式共享内存, DPX 指令, TMA |
| **B100** | 下一代 Tensor Core, 第二代 Transformer Engine, FP4 数据格式, 解压缩引擎 (Decompression Engine) |

> 对程序员的启示：GPU 编程正在从 CUDA 向更专用的编程模式转变。

### 9.2 A100 GPU 架构

**GA100 GPU 中的每个 SM (Streaming Multiprocessor)**：

```
每个 SM 包含：
├── 64 个 FP32 ALU（乘加单元）
├── 32 个 INT32 ALU
└── 4 个 "Tensor Core"
    └── 执行 8x4 × 4x8 矩阵乘加指令
        D = A × B + D
        A, B 存储为 FP16，累加器 D 使用 FP32
```

**整芯片规格（GA100）**：
- 108 个 SM 核心
- 6,912 个 FP32 乘加 ALU
- 432 个 Tensor Core
- 最大时钟频率：1.4 GHz
- **19.5 TFLOPS FP32**
- **312 TFLOPS FP16/F32 混合精度（Tensor Core）**

### 9.3 H100 GPU (2022) 架构

H100 关键特性：
- **第四代 Tensor Core**
- **Tensor Memory Accelerator (TMA) 单元**：专用数据移动硬件
- CUDA Cluster 能力
- **HBM3 内存**（最高 80 GB）
- **TSMC 4nm 工艺**
- **800 亿晶体管**

### 9.4 H100 层级结构

H100 的 CUDA、计算和内存三级层次：

| CUDA 层级 | 计算层级 | 内存层级 |
|-----------|----------|---------|
| **Grid** | GPU | 80 GB HBM / 50 MB L2 |
| **Cluster** | CPC (Compute Partition Cluster) | — |
| **Thread Block** | SM | 256 KB 共享内存/SM |
| **Threads** | SIMD Lanes | 1 KB RF/线程，64 KB/SM 分区 |

- Thread Block Cluster：最多 16 个线程块的集合
- 每个线程块保证在独立的 SM 上执行，且同时运行

### 9.5 H100 SM 内部结构

```
单个 SM 内部：
┌─────────────────────────────────────────────┐
│  共享内存 / L1 缓存：256 KB                    │
├─────────────────────────────────────────────┤
│  4 个 Warp Selector + Fetch/Decode 单元        │
│  (每个时钟 1 个 warp)                          │
├─────────────────────────────────────────────┤
│  SIMD FP32 单元：16个单元共享控制，32-wide      │
│  (每时钟 32x MUL-ADD)                          │
│  SIMD INT 单元：16个单元共享控制                │
│  (每2时钟 16x MUL/ADD)                         │
│  SIMD FP64 单元：8个单元共享控制                │
│  (每2时钟 16x MUL/ADD)                         │
│  加载/存储单元                                  │
├─────────────────────────────────────────────┤
│  4 个 Tensor Core 单元                         │
│  每 TC：16x16x16 [fp16 × fp16 → fp32]          │
├─────────────────────────────────────────────┤
│  Tensor Memory Accelerator (TMA)              │
├─────────────────────────────────────────────┤
│  寄存器：每 SM 总共 256 KB                      │
│  每子核心 64 KB，分配给最多 64 个 Warp            │
└─────────────────────────────────────────────┘
```

### 9.6 Tensor Memory Accelerator (TMA)

TMA 是 H100 引入的**专用数据移动硬件**：

- **异步**加载/存储张量的一个区域（全局内存 ↔ 共享内存）
- 通过**拷贝描述符 (Copy Descriptor)** 描述数据区域
- **单线程**发起 TMA 操作（`cuda::memcpy_async`）
- 拷贝完成时通知 **barrier**
- 硬件自动进行地址生成和数据移动

### 9.7 H100 计算能力总结

| 计算类型 | 性能 |
|----------|------|
| Tensor Core (Systolic Array MMA) FP16 | **989 TFLOPS** |
| SIMD FP16 | 134 TFLOPS |
| SIMD FP32 | 67 TFLOPS |

**关键洞察**：GPU 总 TFLOPS 能力的绝大部分集中在 Tensor Core 中，而非传统 SIMD 单元。

### 9.8 Tensor Core 占比趋势

| GPU | Tensor Core 在总 TFLOPS 中的占比 |
|-----|-------------------------------|
| V100 | ~89% |
| A100 | ~94% |
| H100 | ~96% |
| B100 | ~98% |

趋势：GPU 正从通用并行处理器演变为**以专用矩阵加速为核心**的领域加速器。

### 9.9 B100 的 Tensor Core 编程

B100 进一步远离传统 CUDA 编程模型：

```
B100 Tensor Core 编程步骤：
1. 分配 TMEM 和描述符 → tcgen05.alloc
2. 用 TMA 异步预取/流式传输 tiles → cp.async.bulk.tensor
   (与 mbarrier 协调)
3. 启动异步 MMA → tcgen05.mma batch + tcgen05.commit
4. 排序和退役 → tcgen05.fence
```

- 不再使用 warp 模式
- 单线程执行 MMA
- 张量数据存放于 SMEM 和 TMEM（不再仅仅是寄存器）

> "Not your father's CUDA" — B100 的编程模型已与经典 CUDA 截然不同。

## 10. Google TPU 与脉动阵列

### 10.1 TPU v1 架构特征

Google 的 Tensor Processing Unit (TPU v1) 是深度学习加速的里程碑：

**面积分布**：
- 算术单元仅占约 **30%** 芯片面积
- 控制逻辑面积占比极低

**关键指令**：

```
TPU v1 指令集：
├── read_host_memory      // 从主机读取数据
├── write_host_memory     // 写回主机
├── read_weights          // 读取权重
├── matrix_multiply / convolve  // 核心矩阵/卷积运算
└── activate              // 激活函数
```

> 参考：Jouppi et al. 2017

### 10.2 TPU 的设计哲学

与通用 CPU/GPU 的根本区别：
- **极简的指令集**：仅支持深度学习推理必需的少数操作
- **极简的控制逻辑**：将绝大多数芯片面积用于计算和数据路径
- **脉动阵列 (Systolic Array)** 作为核心计算引擎

## 11. 脉动阵列详解

### 11.1 什么是脉动阵列

脉动阵列是一种**数据驱动的二维计算网格**，专门高效执行矩阵运算。

核心组件：
- **PE (Processing Element)**：阵列中的处理单元
- **Accumulators (32-bit)**：累加器，保存部分和
- **Weights FIFO**：权重先入先出队列

### 11.2 矩阵-向量乘法示例 (y = Wx)

考虑 4x4 权重矩阵 W 与 4x1 输入向量 x 的乘法（y = Wx）：

```
W = [w00 w01 w02 w03]     x = [x0]
    [w10 w11 w12 w13]         [x1]
    [w20 w21 w22 w23]         [x2]
    [w30 w31 w32 w33]         [x3]
```

**脉动阵列工作过程**：（4x4 PE 网格）

**阶段 1**：
```
x0 进入第 1 列 PE → 计算 y0 = x0*w00
   (开始流水)
```

**阶段 2**：
```
x1 进入第 1 列 PE, x0 传递到第 2 列 PE
   y0 += x1*w01 (第1列); y1 = x0*w10 (第2列)
```

**阶段 3**：
```
x2 进入第 1 列 PE, x1→第2列, x0→第3列
   y0 += x2*w02 (第1列); y1 += x1*w11 (第2列)
   y2 = x0*w20 (第3列)
```

**阶段 4（最终输出）**：
```
y0 = x0*w00 + x1*w01 + x2*w02 + x3*w03  (第1列PE输出)
y1 = x0*w10 + x1*w11 + x2*w12 + x3*w13  (第2列PE输出)
y2 = x0*w20 + x1*w21 + x2*w22 + x3*w23  (第3列PE输出)
y3 = x0*w30 + x1*w31 + x2*w32 + x3*w33  (第4列PE输出)
```

### 11.3 矩阵-矩阵乘法 (Y = W X)

对于更大规模的矩阵乘法，使用同样的脉动原则：

```
W(4x4) × X(4xN) = Y(4xN)

关键点：需要多个 4x32bit 累加器来保存输出矩阵的列
```

输入矩阵 X 的列按流水线顺序流入 PE 网格，权重从 FIFO 流中持续加载。

### 11.4 构建更大的矩阵乘法

示例：A = 8x8, B = 8x4096, C = 8x4096

- 使用 4096 个累加器
- A 矩阵按 4 列分块逐步送入
- 每个分块与相应 B 行进行乘累加
- 最终在累加器中得到完整的 C 矩阵

## 12. SIMD vs. 脉动阵列对比

| 特征 | SIMD | 脉动阵列 |
|------|------|---------|
| **数据流** | 控制驱动（指令） | 数据驱动（波前/wavefront） |
| **局部性（数据重用）** | 有限 | 时间 + 空间双重局部性 |
| **通信** | 全局（寄存器/内存） | 局部（相邻 PE 间） |
| **控制** | 集中式 | 分布式 |
| **效率 (perf/mm², perf/Watt)** | 中等 | **极高** |

### 关键区别分析

1. **数据驱动 vs. 控制驱动**：脉动阵列中没有传统的指令流；数据到达即触发计算
2. **局部通信**：数据仅在相邻 PE 之间传递，极大减少了全局互连的能耗
3. **分布式控制**：每个 PE 有简单的本地控制，无需复杂的全局指令调度
4. **空间 + 时间局部性**：权重停留（空间局部性）的同时，输入数据流过（时间局部性利用）

## 13. TPU 性能与演进

### 13.1 TPU v1 性能/功耗

TPU v1 相比同时代 CPU 和 GPU 的性能优势（Jouppi et al. 2017）：

- **GM (Geometric Mean)**：所有应用性能的几何平均
- **WM (Weighted Mean)**：按应用权重加权平均
- **Total**：包含主机 CPU 成本
- **Incremental**：仅 TPU 增量的成本

TPU 在推断任务上展现出远超 CPU 和 GPU 的 perf/watt 指标。

### 13.2 Google TPU 的演进

从 TPU v1 到后续版本：
- **TPU v2/v3**：支持训练，更大规模的脉动阵列
- **TPU v4**：进一步优化的光互连、更高带宽
- 每一代都在架构和工艺上持续优化

### 13.3 AI 领域的硬件彩票 (Hardware Lottery)

> "当某个研究方向之所以胜出，是因为它恰好适**配了现有的软件和硬件**，而非因为该方向在本质上优于替代研究路线时，就出现了**硬件彩票**现象。"
>
> — Sara Hooker

**硬件彩票的循环**：
```
TPU/Dense MM 成功
    ↓
设计更专用化的硬件（更高的 OI ∝ n）
    ↓
Transformer 模型受益最大
    ↓
Transformer 模型统治 AI 领域
    ↓
为矩阵乘法进一步专用化硬件
    ↓
（循环往复...）
```

这意味着：**当前 AI 硬件的发展方向可能正在塑造（而非仅仅服务）AI 算法的研究方向**。

## 14. 硬件彩票与数据流架构

### 14.1 AI 计算的数据流本质

回忆 AI 模型的计算图是数据流图：
- GEMM（矩阵乘）是关键算子
- 此外还有 Pool、SoftMax、Sum 等并行模式操作
- 并行模式：map, filter, reduce 等

### 14.2 Plasticine：可重构数据流架构

Plasticine（Prabhakar, Zhang, et al. ISCA 2017）是一种**可重构数据流架构**：

```
架构组成：
┌──────────────────────────────────────┐
│  Switch (S) — 交换单元                 │
│  PMU (Pattern Memory Unit) — 模式存储  │
│  PCU (Pattern Compute Unit) — 模式计算  │
└──────────────────────────────────────┘

数据流映射：
AI 模型 (GEMM + Parallel Patterns)
    ↓
数据流图（GEMM + map/filter/reduce）
    ↓
可重构数据流架构上的空间映射
```

### 14.3 可重构数据流 vs. 理想加速器

可重构数据流架构的特点：
- **无指令** → 无指令读取/解码开销
- **极端异步**：无序顺序指令执行
- 计算、内存和通信可以完全重叠

这本质上实现了**理想加速器清单中的所有特性**：
- 分块张量
- 异步计算
- 异步内存访问
- 异步芯片间通信
- 计算单元间通信（融合和流水线）
- 流式数据流

## 15. 数据流架构与 FlashAttention 融合

### 15.1 数据流核融合 (Dataflow Kernel Fusion)

在 GPU 上，FlashAttention 将一个 attention 计算分解为多个内核。在数据流架构上，可以对其进行**空间展开**：

```
FlashAttention 的 Tile 化数据流：

  Q (查询) → KT (键转置) → QKT → Mask → Softmax → Dropout → × V
                                                              ↓
                                                          输出 Tile

每个 tile 在独立的 PCU/PMU 上执行上述管道
```

### 15.2 MetaPipeline 概念

多个 tile 可以以管道化方式同时处理：

```
Tile 0: QKT → Mask → Softmax → Dropout → × V
Tile 1: QKT → Mask → Softmax → Dropout → × V
Tile 2: QKT → Mask → Softmax → Dropout → × V
Tile 3: QKT → Mask → Softmax → Dropout → × V
...
```

每个 tile 的计算在空间上映射到不同的 PCU/PMU 组合，形成**MetaPipeline**。

## 16. 总结

### 16.1 专用 AI 硬件的核心原则

专用硬件用于高效执行关键 DNN 计算的关键设计原则：

1. **大量算术单元**：最大化计算密度

2. **可定制/可配置的数据通路**：
   - 中间数据值直接在处理单元之间传输
   - 通过**在芯片上空间布局计算**来调度执行
   - 支持多粒度级别的数据移动优化

3. **大量片上存储**：快速访问中间结果

4. **脉动阵列作为核心计算原语**：
   - 数据驱动的执行
   - 局部通信（仅相邻 PE）
   - 极高的面积/能效

### 16.2 关键术语总结

| 术语 | 定义 |
|------|------|
| **ASIC** | 专用集成电路，固定功能，最高能效 |
| **FPGA** | 现场可编程门阵列，可重配置逻辑，中高能效 |
| **DSP** | 数字信号处理器，VLIW/SIMD 指令增强 |
| **TPU** | Google 的张量处理单元，脉动阵列为核心 |
| **脉动阵列** | 数据驱动的 2D PE 网格，高效矩阵乘法 |
| **Tensor Core** | NVIDIA GPU 中的矩阵乘加硬件单元 |
| **TMA** | Tensor Memory Accelerator，专用异步数据移动 |
| **数据流架构** | 无指令、空间映射计算 |
| **硬件彩票** | 硬件可用性塑造算法研究方向的现象 |
| **GEMM** | General Matrix Multiply，矩阵乘法的核心模式 |

### 16.3 核心教训

1. **能效是万事之本**：从移动设备到数据中心，能量约束是决定性的
2. **专用化 = 能效**：去除通用性的开销可以换来 100-1000 倍的能效提升
3. **数据移动 > 计算**：在现代系统中，数据移动的能耗远超计算本身
4. **AI 正重新定义计算**：AI 工作负载的兴起正在驱动全新的架构设计和编译技术
5. **硬件与算法相互塑造**：理解"硬件彩票"对于 AI 研究方向的选择至关重要
