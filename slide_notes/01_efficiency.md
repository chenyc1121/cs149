## 1. 什么是并行计算？

### 1.1 定义

**并行计算机**是多个处理单元的集合，这些处理单元通过相互协作来**快速**解决问题。

核心关注点：
- **性能 (Performance)**：程序要跑得快
- **效率 (Efficiency)**：要充分利用硬件资源

### 1.2 课堂演示总结

课程通过三个生动演示揭示了并行计算的关键挑战：

| 演示 | 观察结果 | 核心教训 |
|------|---------|---------|
| Demo 1：2处理器并行求和 | 通信限制了加速比 | 学生（处理器）之间传递部分和产生了通信开销；减少通信成本（靠近站位/允许喊话）可提升加速比 |
| Demo 2：扩展到4处理器 | 工作分配不均限制了加速比 | 部分处理器提前完成工作进入空闲，而其他处理器仍在忙碌；改进工作分配可提升加速比 |
| Demo 3：大规模并行 | 通信成本远超计算成本 | 当通信开销压倒计算时，加速比严重受限 |

## 2. 课程三大主题

### 2.1 主题一：设计和编写可扩展的并行程序

**并行思维三步骤**：
1. **分解工作**：将任务拆分为可安全并行执行的子任务
2. **分配工作**：将子任务分配给处理单元
3. **管理通信/同步**：控制处理器之间的通信和同步，避免其限制加速比

编写并行代码需要通过主流并行编程语言中的抽象和机制来实现。

### 2.2 主题二：并行计算机硬件实现

- 实现抽象所需的机制
- 实现的性能特征
- 设计权衡：性能 vs. 便利性 vs. 成本

**为什么需要了解硬件？**
- 机器特性对程序性能至关重要（回顾演示中通信速度的影响）
- 因为你需要关注效率和性能

### 2.3 主题三：思考效率

**核心观点：快 (FAST) 不等于高效 (EFFICIENT)**

- 程序在并行计算机上运行更快，并不代表它在高效地使用硬件
- 例如：在有10个处理器的计算机上获得2倍加速比，这是好结果吗？
- **程序员视角**：充分利用机器提供的能力
- **硬件设计者视角**：选择正确的系统能力（性能/成本权衡，成本包括芯片面积、功耗等）

## 4. 为什么需要并行计算？

### 4.1 历史背景：为什么曾经避免并行？

在 ~2005 年之前：
- **单线程 CPU 性能每 18 个月翻倍**（摩尔定律带来的红利）
- 结论：花时间并行化代码常常不值得 —— 开发者什么都不做，明年硬件自动让代码跑得更快

### 4.2 性能提升的两大传统来源

1. **指令级并行 (Instruction-Level Parallelism, ILP)**：超标量执行
2. **提高 CPU 时钟频率**

### 4.3 免费午餐的终结

三条曲线说明问题：
- **晶体管密度**：持续增长
- **时钟频率**：停止增长（受功耗限制）
- **指令级并行 (ILP)**：收益递减，已挖尽
- **功耗**：急剧上升（成为关键约束）

> **结论**：单指令流性能增长已趋于零。硬件设计者通过增加并行执行单元来提升性能。软件必须被写成并行的才能获得性能提升 —— 软件开发者的"免费午餐"结束了。

## 5. 计算机程序基础

### 5.1 从一个C程序说起

```c
int main(int argc, char** argv) {
    int x = 1;
    for (int i = 0; i < 10; i++) {
        x = x + x;
    }
    printf("%d\n", x);
    return 0;
}
```

### 5.2 程序是什么？（处理器视角）

**程序 = 一组处理器指令的列表**

C代码经编译后生成汇编指令，例如：
```asm
_main:
100000f10:  pushq   %rbp
100000f11:  movq    %rsp, %rbp
100000f14:  subq    $32, %rsp
100000f18:  movl    $0, -4(%rbp)
100000f1f:  movl    %edi, -8(%rbp)
100000f22:  movq    %rsi, -16(%rbp)
100000f26:  movl    $1, -20(%rbp)       ; x = 1
100000f2d:  movl    $0, -24(%rbp)       ; i = 0
100000f34:  cmpl    $10, -24(%rbp)      ; 比较 i < 10
100000f38:  jge     23 <_main+0x45>     ; 如果 i >= 10，跳出循环
100000f3e:  movl    -20(%rbp), %eax     ;
100000f41:  addl    -20(%rbp), %eax     ; x = x + x
100000f44:  movl    %eax, -20(%rbp)     ;
100000f47:  movl    -24(%rbp), %eax     ;
100000f4a:  addl    $1, %eax            ; i++
100000f4d:  movl    %eax, -24(%rbp)     ;
100000f50:  jmp     -33 <_main+0x24>    ; 跳回循环开始
...
100000f72:  ret
```

### 5.3 关键概念

- **指令 (Instruction)**：描述处理器应执行的操作，执行指令通常会修改计算机的**状态**
- **计算机状态 (State)**：程序数据的值，存储在处理器**寄存器**或**内存**中

## 6. 处理器架构基础

### 6.1 简单处理器模型

```
┌────────────────────────────┐
│   Fetch/Decode             │  确定下一步要运行什么指令
├────────────────────────────┤
│   Execution Context        │
│   ┌──────┬──────┬──────┐   │
│   │ R0   │ R1   │ R2   │   │  寄存器：维护程序状态
│   │ R3   │ ...   │      │   │  存储变量的输入输出值
│   └──────┴──────┴──────┘   │
├────────────────────────────┤
│   ALU (Execution Unit)     │  执行单元：执行指令描述的操作
│                            │  可能修改寄存器或内存的值
└────────────────────────────┘
```

- **寄存器 (Registers)**：维护程序状态，存储操作输入和输出
- **执行单元 (ALU/Execution Unit)**：执行指令描述的算术/逻辑操作
- **取指/译码 (Fetch/Decode)**：确定下一条要执行的指令

### 6.2 指令执行示例：加法

指令 `add R0 ← R0, R1` 的含义：将 R0 和 R1 的内容相加，结果存入 R0

执行步骤：
1. **取指 (Fetch)**：从内存获取下一条指令
2. **读取操作数**：从寄存器获取输入值（R0=32, R1=64）
3. **执行**：ALU 计算 32 + 64 = 96
4. **写回**：将结果 96 存入 R0

### 6.3 简单程序的指令序列

```asm
ld   r0, addr[r1]    ; 从内存加载
mul  r1, r0, r0      ; 乘法
mul  r1, r1, r0      ; 乘法
...                  ; 更多指令
st   addr[r2], r0    ; 存储到内存
```

最简单的处理器每时钟周期执行一条指令。

## 7. 指令级并行 (ILP)

### 7.1 示例：计算平方和

```asm
; a = x*x + y*y + z*z
; 假设 R0=x, R1=y, R2=z
mul R0, R0, R0   ; 1. R0 = x*x
mul R1, R1, R1   ; 2. R1 = y*y
mul R2, R2, R2   ; 3. R2 = z*z
add R0, R0, R1   ; 4. R0 = x*x + y*y
add R3, R0, R2   ; 5. R3 = a
```

**依赖关系分析**：
- 指令 1、2、3 之间无依赖 → 可以并行执行
- 指令 4 依赖指令 1 和 2 → 必须在 1、2 之后
- 指令 5 依赖指令 4 → 必须在 4 之后

**ILP 形状**：
```
        x*x ──┐
        y*y ──┼── a  ← ILP = 3（三个乘法并行）
        z*z ──┘
```

### 7.2 超标量执行 (Superscalar Execution)

**核心思想**：处理器自动（或通过编译器辅助）在指令序列中找到独立的指令，并在多个执行单元上并行执行它们。

```
        时钟周期
        ├─ 1 ──┼─ 2 ──┼─ 3 ──┤
执行单元1: mul R0  │ add R0 │ add R3 │
执行单元2: mul R1  │ (空闲) │ (空闲) │
执行单元3: mul R2  │ (空闲) │ (空闲) │
```

- 指令 1、2、3 在周期 1 同时执行
- 指令 4 在周期 2 执行（等待 1、2 的结果）
- 指令 5 在周期 3 执行（等待 4 的结果）
- 5 条指令只需 3 个周期完成，而不是 5 个

**超标量处理器示意**：
```
         ┌──────────┐
         │Fetch/Decode 1│──┐
         └──────────┘  │
         ┌──────────┐  │
         │Fetch/Decode 2│──┤  乱序控制逻辑
         └──────────┘  │       │
                       ├───┬───┴───┬───┐
                    ┌──┘   │       │   └──┐
                 Exec 1 Exec 2   ...  Exec N
```
每时钟周期可解码并发射最多两条（或多条）指令。

### 7.3 依赖图示例

```c
a = 2
b = 4
tmp2 = a + b        // 6   ← 依赖 a,b
tmp3 = tmp2 + a     // 8   ← 依赖 tmp2,a
tmp4 = b + b        // 8   ← 依赖 b
tmp5 = b * b        // 16  ← 依赖 b
tmp6 = tmp2 + tmp4  // 14  ← 依赖 tmp2,tmp4
tmp7 = tmp5 + tmp6  // 30  ← 依赖 tmp5,tmp6
if (tmp3 > 7)
    print tmp3
else
    print tmp7
```

**指令依赖图**展现了指令间的前后依赖关系：
- `tmp4=b+b` 和 `tmp5=b*b` 与 `tmp2=a+b` 无依赖，可并行
- `tmp3=tmp2+a` 依赖 `tmp2`
- `tmp6=tmp2+tmp4` 依赖 `tmp2` 和 `tmp4`
- `tmp7=tmp5+tmp6` 依赖所有之前的计算

### 7.4 ILP 的收益递减

实验数据（Johnson 1991）表明：
- 大多数可用的 ILP 已被每时钟周期可发射 4 条指令的处理器充分利用
- **构建能发射更多指令的处理器几乎没有额外性能收益**

## 8. 功耗墙 (The Power Wall)

### 8.1 功耗公式

**动态功耗**:

$$\text{Power}_{\text{dynamic}} \propto C \times V^2 \times f$$

其中：
- $C$ = 容性负载 (capacitive load)
- $V$ = 电压 (voltage)
- $f$ = 频率 (frequency)

**静态功耗**：晶体管即使不活动也会因漏电流而消耗功率。

### 8.2 功耗数据

| 处理器/设备 | TDP (热设计功耗) |
|------------|-----------------|
| Intel Core i9 10900K (桌面) | 95W |
| Apple M1 (笔记本) | 13W |
| NVIDIA RTX 4090 GPU | 450W |
| 标准微波炉 | 900W |
| 手机处理器 | 0.5 - 2W |
| Frontier 超级计算机 | ~21 兆瓦 |

### 8.3 最大频率限制

处理器的最大允许频率由核心电压决定。功耗随频率超线性增长（因为提高频率通常也需要提高电压）。

## 11. 内存与缓存

### 11.1 内存基础

- 计算机内存组织为**按字节寻址的数组**
- 每个字节由其在内存中的**地址**（在数组中的位置）标识
- 程序的内存地址空间是一段连续的字节范围

```
地址:  0x0  0x1  0x2  ... 0x8  0x9  0xA  0xB  0xC  ... 0x10 ...
值:    16   255  14  ... 6    32   48   255  255 ... 128  ...
```

### 11.2 加载指令 (Load)

```asm
ld R0 ← mem[R2]
```
含义：从寄存器 R2 存储的地址处，加载内存中的四字节值，放入寄存器 R0。

### 11.3 内存访问延迟

**内存访问延迟 (Memory Access Latency)**：内存系统向处理器提供数据所需的时间。

- 示例：100 个时钟周期，100 纳秒
- 内存访问延迟是导致处理器**停顿 (Stall)** 的主要原因

### 11.4 停顿 (Stalls)

当处理器因为后续指令依赖尚未完成的前序指令而无法继续执行时，发生**停顿**。

```asm
ld r0, mem[r2]    ; 加载操作，延迟大
ld r1, mem[r3]    ; 加载操作，延迟大
add r0, r0, r1    ; 依赖 r0 和 r1，必须等待它们加载完成
```

**依赖关系**：`add` 指令只有在 `mem[r2]` 和 `mem[r3]` 的数据加载完成后才能执行。
内存访问时间约 **几百个时钟周期**。

### 11.5 缓存 (Cache)

**定义**：缓存是硬件实现细节，不影响程序输出，只影响性能。缓存是片上存储，保存内存中部分值的副本。

```
DRAM (主存)
   ↕
┌──────────────┐
│  Data Cache  │
│ Line  | Values │
│ 0x4   | 0 0 6 0│
│ 0xC   |255 0 0 0│
└──────┬───────┘
       ↕
┌──────────────┐
│  Processor   │
└──────────────┘
```

**缓存特性**：
- 缓存以**缓存行 (Cache Line)** 的粒度操作（例如每行 4 字节）
- 图中缓存容量为 2 行，每行 4 字节
- 如果地址在缓存中，处理器访问速度远快于 DRAM

### 11.6 缓存替换策略

**LRU (Least Recently Used，最近最少使用)**：当需要腾出空间存放新数据时，丢弃缓存中访问时间最久远的数据。

相关术语（建议自学）：
- 直接映射缓存 (Direct Mapped Cache)
- 组相联缓存 (Set-Associative Cache)
- 缓存行 (Cache Line)

### 11.7 缓存示例一：局部性命中

访问序列：`0x1, 0x2, 0x3, 0x2, 0x1` （假设缓存容量 8 字节，4 字节/行，2 行，LRU 策略）

- **冷缺失 (Cold Miss)**：访问 0x0 → 加载行 0x0
- **空间局部性 (Spatial Locality)**：加载一行缓存会自动"预加载"同一行中其他地址的数据，后续访问不同地址时产生缓存命中
- **时间局部性 (Temporal Locality)**：反复访问同一地址导致命中

### 11.8 缓存示例二：容量缺失

访问序列：`0x1, 0x2, 0x3, 0x4, 0x5, ..., 0xF, 0x0` （顺序遍历 16 字节数组）

当缓存容量不足以容纳所有访问地址时：
- 访问 0x8 → **冷缺失**，加载行 0x8，驱逐行 0x0
- 访问 0xC → **冷缺失**，加载行 0xC，驱逐行 0x4
- 回到 0x0 → **容量缺失 (Capacity Miss)**，需重新加载行 0x0，驱逐行 0x8

### 11.9 缓存层次结构

现代计算机中，线性内存地址空间的实现是复杂的层次结构：

```
┌────────────┐
│ Processor  │
└─────┬──────┘
      ↓  (最快, 最小)
┌────────────┐
│ L1 Cache   │  32 KB,  延迟 ~4 周期 (@4GHz)
├────────────┤
│ L2 Cache   │  256 KB, 延迟 ~12 周期
├────────────┤
│ L3 Cache   │  20 MB,  延迟 ~38 周期
├────────────┤
│ DRAM       │  64 GB,  延迟 ~248 周期 (最佳情况)
└────────────┘
      (最慢, 最大)
```

**核心规律**：离处理器越近的缓存容量越小但延迟越低；离处理器越远的缓存容量越大但延迟越高。

### 11.10 数据移动的能耗

| 操作 | 能耗（大约） |
|------|------------|
| 整数运算 | ~1 pJ |
| 浮点运算 | ~20 pJ |
| 从片上小型 SRAM 读取 64 bits (1mm 距离) | ~26 pJ |
| 从低功耗移动 DRAM (LPDDR) 读取 64 bits | ~1200 pJ |

**启示**：
- 以 10 GB/s 从内存读取数据功耗约 1.6W
- 移动 GPU 总功耗预算仅约 1W
- iPhone 6 电池约 7 Wh
- **利用局部性 (locality) 至关重要！**

> **现代系统设计核心法则**：始终寻求减少计算机中的数据移动量。

## 12. 性能度量核心公式

### 12.1 加速比 (Speedup)

$$\text{speedup}(P) = \frac{\text{Execution Time (1 processor)}}{\text{Execution Time (P processors)}}$$

- 使用并行处理的主要动机：获得加速比
- 理想情况下，P 个处理器应获得 P 倍的加速比（线性加速）
- 实际上受限于通信开销、负载不均衡、串行部分等

### 12.2 动态功耗公式

$$\text{Power}_{\text{dynamic}} \propto C \times V^2 \times f$$

其中 $C$ 为容性负载，$V$ 为电压，$f$ 为频率。

总功耗 = 动态功耗 + 静态功耗（漏电流）

### 12.3 效率思考

**FAST != EFFICIENT（快不等于高效）**

评估并行效果时需同时考虑：
- **加速比**：程序是否更快？
- **效率**：硬件资源是否被充分利用？

例如，在 10 核处理器上获得 2x 加速比，加速比为 2，效率仅为 20%。

## 13. 总结

### 核心信息

1. **单线程性能增长极其缓慢** —— 要让程序显著更快，必须利用多个处理单元或专用处理硬件。这意味着你需要懂得如何编写并行且高效的代码。

2. **编写并行程序具有挑战性** —— 需要：
   - 问题分解 (Problem Partitioning)
   - 通信管理 (Communication)
   - 同步控制 (Synchronization)
   - 理解机器特性（尤其是数据移动）

3. **现代计算机拥有远超你想象的强大处理能力** —— 关键在于**高效地使用它**。

4. **实现高效处理几乎总是归结为高效地访问数据。**

### 关键术语

| 术语 | 含义 |
|------|------|
| 加速比 (Speedup) | 并行执行时间相对于串行执行时间的比值 |
| 效率 (Efficiency) | 加速比 / 处理器数量 |
| ILP (指令级并行) | 在单指令流中同时执行多条独立指令 |
| 超标量 (Superscalar) | 处理器自动发现并并行执行独立指令的能力 |
| 功耗墙 (Power Wall) | 功耗成为处理器频率提升的主要限制 |
| 缓存 (Cache) | 片上高速存储，保存主存中部分数据的副本 |
| 空间局部性 (Spatial Locality) | 访问一个地址后可能访问其附近地址 |
| 时间局部性 (Temporal Locality) | 访问一个地址后可能在近期再次访问 |
| 停顿 (Stall) | 处理器因等待数据而无法继续执行 |
| 延迟 (Latency) | 从请求数据到数据可用的时间 |
| 缓存缺失 (Cache Miss) | 请求的数据不在缓存中，需从更慢的存储层级获取 |
